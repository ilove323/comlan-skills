# /debug

> 如遇到不熟悉的占位符或需要查看哪些工具已接入，请参阅 [CONNECTORS.md](../../CONNECTORS.md)。

通过结构化调试会话系统性地定位和修复问题。

## 使用方式

```
/debug $ARGUMENTS
```

## 工作原理

```
┌─────────────────────────────────────────────────────────────────┐
│                       调试                                        │
├─────────────────────────────────────────────────────────────────┤
│  第 1 步：复现                                                    │
│  ✓ 理解预期行为与实际行为                                         │
│  ✓ 确定精确的复现步骤                                             │
│  ✓ 确定范围（什么时候开始的？影响哪些用户？）                      │
│                                                                   │
│  第 2 步：定位                                                    │
│  ✓ 缩小问题所在的组件、服务或代码路径                              │
│  ✓ 检查近期变更（部署、配置变更、依赖更新）                        │
│  ✓ 查看日志和错误信息                                             │
│                                                                   │
│  第 3 步：诊断                                                    │
│  ✓ 形成假设并验证                                                 │
│  ✓ 追踪代码路径                                                   │
│  ✓ 找出根本原因（而不仅是症状）                                    │
│                                                                   │
│  第 4 步：修复                                                    │
│  ✓ 提出修复方案及说明                                             │
│  ✓ 考虑副作用和边界情况                                           │
│  ✓ 建议防止回归的测试                                             │
└─────────────────────────────────────────────────────────────────┘
```

## 我需要的信息

告诉我问题是什么。以下任何信息都有帮助：
- 错误信息或堆栈跟踪
- 复现步骤
- 近期发生了什么变更
- 日志或截图
- 预期行为 vs 实际行为

## 输出格式

```markdown
## 调试报告：[问题摘要]

### 问题复现
- **预期**：[应该发生什么]
- **实际**：[实际发生了什么]
- **步骤**：[如何复现]

### 根本原因
[解释 bug 为何发生]

### 修复方案
[所需的代码变更或配置修复]

### 预防措施
- [需要添加的测试]
- [需要增加的防护]
```

## 如果已接入连接器

如果 **~~monitoring** 已接入：
- 拉取问题发生时间段附近的日志、错误率和指标
- 展示可能与此相关的近期部署和配置变更

如果 **~~source control** 已接入：
- 识别涉及受影响代码路径的近期提交和 PR
- 检查问题是否与某次具体变更相关

如果 **~~project tracker** 已接入：
- 搜索相关 bug 报告或已知问题
- 找到修复方案后创建工单

## 提示

1. **原文提供错误信息** — 不要转述。精确的文本很重要。
2. **说明发生了什么变更** — 近期部署、依赖更新和配置变更是主要嫌疑对象。
3. **提供上下文** — "在测试环境正常但生产环境失败"或"只影响大载荷"能快速缩小范围。
