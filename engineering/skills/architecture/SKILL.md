# /architecture

> 如遇到不熟悉的占位符或需要查看哪些工具已接入，请参阅 [CONNECTORS.md](../../CONNECTORS.md)。

创建架构决策记录（ADR）或评估系统设计。

## 使用方式

```
/architecture $ARGUMENTS
```

## 模式

**创建 ADR**："我们的事件总线应该用 Kafka 还是 SQS？"
**评估设计**："评审这个微服务提案"
**系统设计**："为我们的应用设计通知系统"

关于需求收集、可扩展性分析和权衡评估的详细框架，见 **system-design** 技能。

## 输出格式——ADR

```markdown
# ADR-[编号]：[标题]

**状态：** 已提议 | 已接受 | 已废弃 | 已取代
**日期：** [日期]
**决策者：** [谁需要确认]

## 背景
[什么情境？存在哪些制约因素？]

## 决策
[我们提议做什么改变？]

## 已考虑的方案

### 方案 A：[名称]
| 维度 | 评估 |
|-----------|------------|
| 复杂度 | [低/中/高] |
| 成本 | [评估] |
| 可扩展性 | [评估] |
| 团队熟悉度 | [评估] |

**优点：** [列表]
**缺点：** [列表]

### 方案 B：[名称]
[相同格式]

## 权衡分析
[方案间的关键权衡，附清晰推理]

## 后果
- [什么变得更容易]
- [什么变得更难]
- [将来需要重新审视的内容]

## 行动项
1. [ ] [实施步骤]
2. [ ] [后续跟进]
```

## 提示

1. **提前说明约束** — "我们需要在 2 周内交付"或"必须支撑 10K rps"会直接影响建议方向。
2. **命名你的方案** — 即使你已经有倾向，明确列出替代方案能让分析更加平衡。
3. **包含非功能性需求** — 延迟、成本、团队技术栈熟悉度和维护负担与功能特性同等重要。
