# OpenClaw 插件定制

为特定组织定制插件——既可以首次设置通用插件模板，也可以对已配置的插件进行调整和优化。

> **查找插件**：要查找插件的源文件，运行 `find mnt/.local-plugins mnt/.plugins -type d -name "*<plugin-name>*"` 定位插件目录，然后读取其文件以了解结构，再进行更改。如果找不到插件目录，用户可能正在远程容器中运行此对话。中止操作并告知用户："插件定制目前仅在桌面应用的OpenClaw模式下可用。"

## 确定定制模式

找到插件后，检查`~~`前缀占位符：`grep -rn '~~\w' /path/to/plugin --include='*.md' --include='*.json'`

> **默认规则**：如果存在`~~`占位符，默认进入**通用插件设置**模式，除非用户明确要求定制插件的特定部分。

**1. 通用插件设置** — 插件包含`~~`前缀占位符。这些是模板中需要替换为实际值的定制点（例如 `~~Jira` → `Asana`，`~~your-team-channel` → `#engineering`）。

**2. 局部定制** — 不存在`~~`占位符，且用户要求定制插件的特定部分（例如"定制连接器"、"更新站会技能"、"更换工单工具"）。读取插件文件找到相关部分，仅专注于这些部分。不要扫描整个插件或呈现无关的定制项。

> **旧版 `commands/` 目录**：部分插件包含 `commands/` 目录。OpenClaw界面现在将这些与技能一起呈现为单一的"技能"概念，因此在定制时将 `commands/*.md` 文件与 `skills/*/SKILL.md` 文件同等对待。

**3. 整体定制** — 不存在`~~`占位符，且用户希望广泛修改插件。读取插件文件以了解当前配置，然后询问用户想更改哪些内容。

> **重要**：永远不要更改被定制的插件或技能的名称。不要重命名目录、文件或插件/技能名称字段。

> **非技术性输出**：所有面向用户的输出（待办事项、问题、摘要）必须使用简明的非技术语言撰写。永远不要向用户提及`~~`前缀、占位符或定制点。一切都应以插件功能和组织工具为框架来表述。

## 定制工作流

### 阶段0：收集用户意图（仅限局部定制和整体定制）

对于**局部定制**和**整体定制**（不包括通用插件设置），检查用户是否在请求中提供了自由描述的上下文（例如"定制站会技能——我们每天早晨在#eng-updates进行异步站会"）。

- **若用户提供了上下文**：记录下来并在阶段3中用于预填答案——跳过用户已在此回答的问题。
- **若用户未提供上下文**：在继续之前使用AskUserQuestion提问一个开放性问题。针对其请求定制的内容提问——例如"你对这个简报技能有什么修改设想？"或"你希望改变这个插件的哪些工作方式？"保持简短且与其请求相关。

  将其回答（如有）作为后续阶段的补充背景信息使用。

### 阶段1：从知识MCP收集上下文

使用公司内部知识MCP收集与定制范围相关的信息。详细的各类别查询模式请参见 `references/search-strategies.md`。

**需要收集的内容**（根据相关性选择范围）：

- 组织使用的工具名称和服务
- 组织流程和工作方式
- 团队惯例（命名规范、状态定义、估算刻度）
- 配置值（工作区ID、项目名称、团队标识符）

**可搜索的来源：**

1. **Chat/Slack MCP** — 工具提及、集成信息、工作流讨论
2. **文档MCP** — 入职文档、工具指南、配置说明
3. **邮件MCP** — 许可证通知、管理员邮件、配置邀请

记录所有发现，在阶段3中使用。

### 阶段2：创建待办列表

创建适当范围的待办变更列表：

- **对于局部定制**：仅包含与用户所请求特定部分相关的项目。
- **对于通用插件设置**：运行 `grep -rn '~~\w' /path/to/plugin --include='*.md' --include='*.json'` 查找所有占位符定制点，按主题分组。
- **对于整体定制**：读取插件文件，了解当前配置，根据用户请求确定需要变更的内容。

使用专注于插件目的、对用户友好的描述：

- **好**："了解公司的站会准备方式"
- **差**："替换 skills/standup-prep/SKILL.md 中的占位符"

### 阶段3：完成待办事项

使用阶段0和阶段1的上下文逐项完成。

**若用户的自由描述输入（阶段0）或知识MCP（阶段1）提供了明确答案**：直接应用，无需确认。

**否则**：使用AskUserQuestion。不要假设"行业标准"默认值是正确的——如果用户输入和知识MCP都没有提供具体答案，就直接询问。注意：AskUserQuestion始终包含"跳过"按钮和自由文本输入框，因此不要将"无"或"其他"作为选项。

**变更类型：**

1. **占位符替换**（通用设置）：`~~Jira` → `Asana`，`~~your-org-channel` → `#engineering`
2. **内容更新**：修改说明、技能、工作流或引用以匹配组织实际情况
3. **URL模式更新**：`tickets.example.com/your-team/123` → `app.asana.com/0/PROJECT_ID/TASK_ID`
4. **配置值**：工作区ID、项目名称、团队标识符

若用户不知道或跳过，保持值不变（对于通用设置则保留`~~`前缀占位符）。

### 阶段4：查找有用的MCP

解决所有定制项后，为已识别或变更的工具连接MCP。完整的工作流、类别关键词映射和配置文件格式请参见 `references/mcp-servers.md`。

对于定制过程中识别的每个工具：

1. 搜索注册表：`search_mcp_registry(keywords=[...])` 使用来自 `references/mcp-servers.md` 的类别关键词，若工具名称已知则直接搜索
2. 若未连接：`suggest_connectors(directoryUuids=["chosen-uuid"])` — 用户完成身份验证
3. 更新插件的MCP配置文件（检查 `plugin.json` 中的自定义位置，否则使用根目录的 `.mcp.json`）

收集所有MCP结果，并在摘要输出中一并呈现（见下方）——不要在此阶段逐一呈现MCP。

## 打包插件

应用所有定制后，将插件打包为 `.plugin` 文件提供给用户：

1. **压缩插件目录**（排除 `setup/` 目录，因为已不再需要）：
   ```bash
   cd /path/to/plugin && zip -r /tmp/plugin-name.plugin . -x "setup/*" && cp /tmp/plugin-name.plugin /path/to/outputs/plugin-name.plugin
   ```
2. **以 `.plugin` 扩展名将文件呈现给用户**，以便其直接安装。（呈现 .plugin 文件会向用户显示富预览，用户可以浏览插件文件，并通过点击按钮接受定制结果。）

> **重要**：始终先在 `/tmp/` 中创建zip，然后再复制到输出文件夹。直接写入输出文件夹可能因权限问题失败，并留下临时文件。

> **命名**：使用原始插件目录名作为 `.plugin` 文件名（例如，若插件目录为 `coder`，输出文件应为 `coder.plugin`）。定制期间不要重命名插件或其文件——只替换占位符值和更新内容。

## 摘要输出

定制完成后，向用户呈现按来源分组的信息摘要。始终包含MCP部分，说明设置期间连接了哪些MCP，以及用户还需要连接哪些：

```markdown
## 来自Slack搜索

- 你们使用Asana进行项目管理
- 冲刺周期为2周

## 来自文档搜索

- 故事点使用T恤尺码

## 来自你的回答

- 工单状态为：待办、进行中、审核中、已完成
```

然后呈现设置期间已连接的MCP，以及用户还应连接的MCP，并提供连接说明。

如果阶段1中没有可用的知识MCP，且用户至少手动回答了一个问题，在结尾加上说明：

> 顺便提一下，连接Slack或Microsoft Teams等来源，下次定制插件时我就能自动查找答案了。
