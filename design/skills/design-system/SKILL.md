# /design-system

> 如果您看到不熟悉的占位符或需要检查已连接的工具，请参阅 [CONNECTORS.md](../../CONNECTORS.md)。

管理您的设计系统——审计一致性、记录组件或设计新模式。

## 用法

```
/design-system audit                    # 全系统审计
/design-system document [component]     # 记录组件
/design-system extend [pattern]         # 设计新组件或模式
```

## 设计系统的组成部分

### 设计令牌
定义视觉语言的原子值：
- 颜色（品牌色、语义色、中性色）
- 排版（比例、字重、行高）
- 间距（比例、组件内边距）
- 边框（圆角、宽度）
- 阴影（层级）
- 动效（时长、缓动）

### 组件
具有以下定义的可复用 UI 元素：
- 变体（主要、次要、幽灵）
- 状态（默认、悬停、激活、禁用、加载、错误）
- 尺寸（小、中、大）
- 行为（交互、动画）
- 无障碍性（ARIA、键盘）

### 模式
组合组件的常见 UI 解决方案：
- 表单（输入组、验证、提交）
- 导航（侧边栏、标签、面包屑）
- 数据展示（表格、卡片、列表）
- 反馈（提示、模态框、内联消息）

## 原则

1. **一致性优于创意** — 系统存在的意义是让团队不需要重复发明轮子
2. **约束中的灵活性** — 组件应可组合，而非僵硬
3. **记录一切** — 未记录的即不存在
4. **版本化并迁移** — 破坏性变更需要迁移路径

## 连接器可用时

如果已连接 **~~design tool**：
- 直接在 Figma 中审计组件——检查命名、变体和令牌使用
- 提取组件属性和图层结构用于文档记录

如果已连接 **~~knowledge base**：
- 搜索现有组件文档和使用指南
- 将更新后的文档发布到您的 Wiki

## 使用技巧

1. **从审计开始** — 在决定去哪里之前先了解现状。
2. **边建设边记录** — 在设计组件时记录更容易。
3. **优先覆盖率而非完美** — 80% 的组件有文档胜过 10 个组件有 100% 的文档。
