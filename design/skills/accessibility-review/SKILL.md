# /accessibility-review

对设计或页面进行 WCAG 2.1 AA 无障碍合规审计。

## 用法

```
/accessibility-review $ARGUMENTS
```

审计无障碍性：@$1

## WCAG 2.1 AA 快速参考

### 可感知
- **1.1.1** 非文本内容有替代文本
- **1.3.1** 信息和结构通过语义传达
- **1.4.3** 对比度比 >= 4.5:1（普通文本），>= 3:1（大文本）
- **1.4.11** 非文本对比度 >= 3:1（UI 组件、图形）

### 可操作
- **2.1.1** 所有功能可通过键盘使用
- **2.4.3** 逻辑焦点顺序
- **2.4.7** 可见焦点指示器
- **2.5.5** 触控目标 >= 44x44 CSS 像素

### 可理解
- **3.2.1** 聚焦时可预测（无意外变化）
- **3.3.1** 错误识别（描述错误）
- **3.3.2** 输入框有标签或说明

### 健壮
- **4.1.2** 所有 UI 组件有名称、角色和值

## 常见问题

1. 色彩对比度不足
2. 表单缺少标签
3. 交互元素无法通过键盘访问
4. 有意义的图片缺少替代文本
5. 模态框中的焦点陷阱
6. 缺少 ARIA 地标
7. 自动播放媒体没有控件
8. 有时间限制但无延期选项

## 测试方法

1. 自动化扫描（发现约 30% 的问题）
2. 纯键盘导航
3. 屏幕阅读器测试（VoiceOver、NVDA）
4. 色彩对比度验证
5. 缩放至 200%——布局是否崩溃？

## 连接器可用时

如果已连接 **~~design tool**：
- 直接从 Figma 检查颜色值、字号和触控目标
- 在设计规格中检查组件的 ARIA 角色和键盘行为

如果已连接 **~~project tracker**：
- 为每个无障碍发现创建工单，附带严重程度和 WCAG 准则
- 将发现与现有无障碍修复 Epic 关联

## 使用技巧

1. **从对比度和键盘开始** — 这能发现最常见且影响最大的问题。
2. **使用真实辅助技术测试** — AI助手的审计是个好起点，但用 VoiceOver/NVDA 进行手动测试能发现我发现不了的问题。
3. **按影响优先排序** — 先修复阻碍用户的问题，再打磨细节。
