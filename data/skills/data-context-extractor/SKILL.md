# 数据上下文提取器

一个元技能，从分析师处提取公司专属的数据知识，并生成定制化的数据分析技能。

## 工作原理

该技能有两种模式：

1. **引导模式**：从零开始创建新的数据分析技能
2. **迭代模式**：通过添加特定领域的参考文件来改进现有技能

---

## 引导模式

适用场景：用户希望为其数据仓库创建新的数据上下文技能。

### 第一阶段：数据库连接与发现

**第一步：确定数据库类型**

询问："你使用的是哪种数据仓库？"

常见选项：
- **BigQuery**
- **Snowflake**
- **PostgreSQL/Redshift**
- **Databricks**

使用 `~~data warehouse` 工具（查询和 Schema）建立连接。如不明确，检查当前会话中可用的 MCP 工具。

**第二步：探索 Schema**

使用 `~~data warehouse` Schema 工具：
1. 列出可用的数据集/Schema
2. 找出最重要的表（询问用户："分析师查询最频繁的 3-5 张表是哪些？"）
3. 获取这些关键表的 Schema 详情

各方言示例探索查询：
```sql
-- BigQuery：列出数据集
SELECT schema_name FROM INFORMATION_SCHEMA.SCHEMATA

-- BigQuery：列出数据集中的表
SELECT table_name FROM `project.dataset.INFORMATION_SCHEMA.TABLES`

-- Snowflake：列出 Schema
SHOW SCHEMAS IN DATABASE my_database

-- Snowflake：列出表
SHOW TABLES IN SCHEMA my_schema
```

### 第二阶段：核心问题（逐一询问）

Schema 发现完成后，通过对话方式逐步询问以下问题（不要一次性全部抛出）：

**实体消歧（关键）**
> "你们这里说的'用户'或'客户'具体指什么？是否存在不同类型？"

注意聆听：
- 多种实体类型（user vs account vs organization）
- 它们之间的关系（1:1、1:多、多:多）
- 连接它们的 ID 字段

**主要标识符**
> "[客户/用户/账户]的主要标识符是什么？同一实体是否有多个 ID？"

注意聆听：
- 主键 vs 业务键
- UUID vs 整数 ID
- 遗留 ID 系统

**关键指标**
> "被问到最多的 2-3 个指标是什么？每个指标是如何计算的？"

注意聆听：
- 精确公式（ARR = 月营收 × 12）
- 每个指标涉及的表/字段
- 时间段约定（近7天、自然月等）

**数据清洗规则**
> "哪些数据必须始终从查询中排除？（测试数据、欺诈、内部用户等）"

注意聆听：
- 始终需要包含的标准 WHERE 子句
- 表示排除的标志字段（is_test、is_internal、is_fraud）
- 需要排除的特定值（status = 'deleted'）

**常见陷阱**
> "新分析师通常在这份数据上会犯哪些错误？"

注意聆听：
- 容易混淆的字段名
- 时区问题
- NULL 处理的特殊情况
- 历史表 vs 当前状态表

### 第三阶段：生成技能

按以下结构创建技能：

```
[company]-data-analyst/
├── SKILL.md
└── references/
    ├── entities.md          # 实体定义与关系
    ├── metrics.md           # KPI 计算方式
    ├── tables/              # 每个领域一个文件
    │   ├── [domain1].md
    │   └── [domain2].md
    └── dashboards.json      # 可选：现有仪表板目录
```

**SKILL.md 模板**：参见 `references/skill-template.md`

**SQL 方言章节**：参见 `references/sql-dialects.md`，并纳入对应的方言说明。

**参考文件模板**：参见 `references/domain-template.md`

### 第四阶段：打包与交付

1. 在技能目录中创建所有文件
2. 打包为 zip 文件
3. 向用户呈现，附上已捕获内容的摘要

## 迭代模式

适用场景：用户已有现有技能，但需要添加更多上下文。

### 第一步：加载现有技能

请用户上传现有技能（zip 或文件夹），或在会话中找到它。

阅读当前的 SKILL.md 和参考文件，了解已记录的内容。

### 第二步：识别缺口

询问："哪个领域或话题需要更多上下文？哪些查询会失败或产生错误结果？"

常见缺口：
- 新数据领域（市场、财务、产品等）
- 缺失的指标定义
- 未记录的表关联关系
- 新术语

### 第三步：针对性发现

针对已识别的领域：

1. **探索相关表**：使用 `~~data warehouse` Schema 工具找出该领域的表
2. **提出针对性问题**：
   - "哪些表用于 [领域] 分析？"
   - "[领域] 的关键指标是什么？"
   - "[领域] 数据有哪些特殊过滤条件或陷阱？"

3. **生成新参考文件**：使用领域模板创建 `references/[domain].md`

### 第四步：更新与重新打包

1. 添加新参考文件
2. 更新 SKILL.md 的"知识库导航"章节，纳入新领域
3. 重新打包技能
4. 向用户呈现更新后的技能

## 参考文件标准

每个参考文件应包含：

### 表文档
- **位置**：完整的表路径
- **描述**：该表包含什么内容，何时使用
- **主键**：如何唯一标识行
- **更新频率**：数据刷新频率
- **关键字段**：包含字段名、类型、描述、注意事项的表格
- **关联关系**：与其他表的关联方式
- **示例查询**：2-3 个常见查询模式

### 指标文档
- **指标名称**：可读名称
- **定义**：通俗易懂的说明
- **公式**：包含字段引用的精确计算方式
- **数据来源**：数据所在的表
- **注意事项**：边界情况、排除规则、陷阱

### 实体文档
- **实体名称**：称呼方式
- **定义**：在业务中代表什么
- **主表**：在哪里可以找到该实体
- **ID 字段**：如何标识它
- **关联关系**：与其他实体的关系
- **常用过滤**：标准排除（内部、测试等）
