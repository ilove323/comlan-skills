# /validate-data - 分享前验证分析

> 如遇到不熟悉的占位符或需要查看哪些工具已接入，请参阅 [CONNECTORS.md](../../CONNECTORS.md)。

在与干系人分享前，对分析的准确性、方法论和潜在偏差进行审查。生成置信度评估和改进建议。

## 使用方式

```
/validate-data <待审查的分析>
```

待审查的分析可以是：
- 对话中的文档或报告
- 文件（Markdown、Notebook、电子表格）
- SQL 查询及其结果
- 图表及其基础数据
- 方法论和发现的描述

## 工作流

### 1. 审查方法论与假设

检查：

- **问题框架**：分析是否在回答正确的问题？问题是否可以有不同解读？
- **数据选择**：使用的表/数据集是否正确？时间范围是否合适？
- **分析群体定义**：分析群体是否定义正确？是否存在非预期的排除？
- **指标定义**：指标定义是否清晰一致？是否与干系人的理解相符？
- **基准与对比**：对比是否公平？时间段、同期群大小和背景是否具有可比性？

### 2. 执行交付前 QA 检查清单

逐项检查以下内容——数据质量、计算、合理性和呈现方式。

### 3. 排查常见分析陷阱

针对以下详细陷阱目录系统性审查：连接爆炸、幸存者偏差、不完整周期对比、分母变动、均值的均值、时区不一致、选择偏差。

### 4. 验证计算与聚合

尽可能进行抽查：

- 独立重新计算几个关键数字
- 验证子总计是否正确累加为总计
- 检查百分比是否合计约为 100%（在预期如此的情况下）
- 确认同比/环比使用了正确的基准期
- 验证过滤器是否在所有指标上一致应用

使用以下结果合理性检验技术：量级检查、交叉验证、危险信号检测。

### 5. 评估可视化

如果分析包含图表：

- 轴是否从合适的值开始（柱状图从零开始）？
- 对比图表的刻度是否一致？
- 图表标题是否准确描述了所呈现的内容？
- 可视化是否可能误导快速阅读者？
- 是否存在截断轴、不一致间隔或扭曲视觉的 3D 效果？

### 6. 评估叙述与结论

审查：

- 结论是否得到所呈现数据的支撑
- 是否承认了其他可能的解释
- 不确定性是否得到适当传达
- 建议是否从发现中合乎逻辑地推导出来
- 置信程度是否与证据强度相符

### 7. 提出改进建议

提供具体、可执行的建议：

- 能强化结论的补充分析
- 应注明的注意事项或局限性
- 关键点的更佳可视化或表达方式
- 干系人会需要的缺失背景

### 8. 生成置信度评估

按三级标准评级：

**可直接分享** — 分析方法健全，计算已验证，注意事项已注明。有少量改进建议，但无阻塞性问题。

**附注意事项分享** — 分析基本正确，但存在特定局限或假设，必须向干系人说明。列出必须传达的注意事项。

**需要修改** — 发现具体错误、方法论问题或缺失分析，应在分享前修正。按优先级列出需要修改的内容。

## 交付前 QA 检查清单

与干系人分享任何分析前，逐项检查以下内容。

### 数据质量检查

- [ ] **来源验证**：确认所用的表/数据来源。它们适合回答这个问题吗？
- [ ] **时效性**：数据对该分析足够新。已注明"数据截至"日期。
- [ ] **完整性**：时间序列或细分中无意外断层。
- [ ] **空值处理**：检查了关键字段的空值率。空值处理方式合理（排除、填充或标记）。
- [ ] **去重**：确认没有因错误关联或重复源数据导致的重复计数。
- [ ] **过滤验证**：所有 WHERE 条件和过滤器均正确。无非预期排除。

### 计算检查

- [ ] **聚合逻辑**：GROUP BY 包含所有非聚合字段。聚合粒度与分析粒度匹配。
- [ ] **分母正确性**：比率和百分比使用了正确的分母。分母非零。
- [ ] **日期对齐**：对比使用了相同的时间段长度。不完整周期已排除或注明。
- [ ] **连接正确性**：JOIN 类型合适（INNER vs LEFT）。多对多连接没有使计数膨胀。
- [ ] **指标定义**：指标与干系人的定义一致。任何偏差均已注明。
- [ ] **子总计求和**：各部分之和符合预期等于整体。若不等，解释原因（如重叠）。

### 合理性检查

- [ ] **量级**：数字在合理范围内。营收非负。百分比在 0-100% 之间。
- [ ] **趋势连续性**：时间序列中无无法解释的突升或骤降。
- [ ] **交叉验证**：关键数字与其他已知来源（仪表板、历史报告、财务数据）匹配。
- [ ] **数量级**：总营收在合理范围内。用户数与已知数据吻合。
- [ ] **边界情况**：在边界处会怎样？空细分、零活跃度周期、新实体。

### 呈现检查

- [ ] **图表准确性**：柱状图从零开始。轴已标注。同类图表刻度一致。
- [ ] **数字格式**：精度适当。货币/百分比格式一致。大数字加千位分隔符。
- [ ] **标题清晰**：标题陈述洞察而非仅描述指标。已说明时间范围。
- [ ] **注意事项透明**：已明确说明已知局限和假设。
- [ ] **可复现性**：其他人可以根据所提供的文档重现此分析。

## 常见数据分析陷阱

### 连接爆炸

**问题**：多对多连接悄无声息地成倍增加行数，使计数和汇总虚高。

**如何检测**：
```sql
-- 连接前后检查行数
SELECT COUNT(*) FROM table_a;  -- 1,000
SELECT COUNT(*) FROM table_a a JOIN table_b b ON a.id = b.a_id;  -- 3,500（有问题）
```

**如何预防**：
- 连接后始终检查行数
- 如果计数增加，调查连接关系（真的是 1:1 还是 1:多？）
- 通过连接计数实体时，用 `COUNT(DISTINCT a.id)` 代替 `COUNT(*)`

### 幸存者偏差

**问题**：只分析当前存在的实体，忽略了那些已被删除、流失或失败的。

**示例**：
- 分析"当前用户"的行为，遗漏了已流失的用户
- 查看"使用我们产品的公司"，忽略了评估后离开的公司
- 研究"成功"结果的特征，不包含"失败"结果

**如何预防**：在得出结论前问"谁不在这个数据集里？"

### 不完整周期对比

**问题**：将不完整的周期与完整周期进行对比。

**示例**：
- "1月营收为 50万，而12月为 80万"——但1月还没结束
- "本周注册量下降"——周三就查看，与上一个完整周对比

**如何预防**：始终过滤为完整周期，或按同月日/相同天数进行对比。

### 分母变动

**问题**：周期之间分母发生变化，使比率不可比。

**示例**：
- 因为改变了"符合资格"用户的计算方式，转化率提升了
- 因为更新了"活跃"的定义，流失率发生了变化

**如何预防**：在所有对比周期使用一致的定义。注明任何定义变更。

### 均值的均值

**问题**：当各组大小不同时，对预先计算的平均值再取平均会得出错误结果。

**示例**：
- A 组：100 个用户，平均营收 50 元
- B 组：10 个用户，平均营收 200 元
- 错误：均值的均值 = （50 + 200）/ 2 = 125 元
- 正确：加权平均 = （100×50 + 10×200）/ 110 ≈ 63.64 元

**如何预防**：始终从原始数据聚合。永远不要对已聚合的均值再取平均。

### 时区不一致

**问题**：不同数据来源使用不同时区，导致数据错位。

**示例**：
- 事件时间戳用 UTC，用户可见日期用本地时间
- 使用不同截止时间的每日汇总

**如何预防**：分析前将所有时间戳标准化为统一时区（推荐 UTC）。注明所用时区。

### 细分中的选择偏差

**问题**：细分由你所测量的结果定义，形成循环逻辑。

**示例**：
- "完成引导的用户留存率更高"——显然，他们是自我选择的
- "高活跃用户创造更多营收"——他们就是靠创造营收成为高活跃用户的

**如何预防**：基于干预前的特征而非结果来定义细分。

### 其他统计陷阱

- **辛普森悖论**：聚合与分细分时趋势反转
- **将相关性呈现为因果关系**，缺乏支撑证据
- **小样本量**导致不可靠的结论
- **异常值过度影响均值**（是否应使用中位数？）
- **多重检验/挑选**显著结果
- **前视偏差**：使用未来信息解释过去事件
- **刻意选择的时间范围**，偏向特定叙事

## 结果合理性检验

### 量级检查

对分析中的任何关键数字，验证是否通过"直觉测试"：

| 指标类型 | 合理性检查 |
|---|---|
| 用户数量 | 是否与已知的 MAU/DAU 相符？ |
| 营收 | 是否与已知 ARR 在同一数量级？ |
| 转化率 | 是否在 0% 到 100% 之间？是否与仪表板数字一致？ |
| 增长率 | 月增长率 50%+ 是否现实，还是存在数据问题？ |
| 均值 | 考虑到已知的分布情况，均值是否合理？ |
| 百分比 | 各细分百分比之和是否约为 100%？ |

### 交叉验证技术

1. **用两种不同方法计算同一指标**，验证结果是否吻合
2. **抽查个别记录**——挑几个特定实体，手动追踪其数据
3. **与已知基准对比**——与已发布的仪表板、财务报告或历史分析进行匹配
4. **反推验证**——如果总营收为X，那么人均营收乘以用户数是否大致等于X？
5. **边界检查**——过滤到单日、单个用户或单个类别时会怎样？这些微观结果合理吗？

### 需要调查的危险信号

- 任何环比变化超过 50% 且无明显原因的指标
- 计数或求和结果恰好是整数（可能存在过滤或默认值问题）
- 比率恰好为 0% 或 100%（可能数据不完整）
- 结果与假设完美吻合（现实通常更加混乱）
- 不同时间段或细分的数值完全相同（可能查询忽略了某个维度）

## 可复现性文档标准

### 分析文档模板

每个非简单分析都应包含：

```markdown
## 分析：[标题]

### 问题
[所回答的具体问题]

### 数据来源
- 表：[schema.table_name]（截至 [日期]）
- 表：[schema.other_table]（截至 [日期]）
- 文件：[文件名]（来源：[获取渠道]）

### 定义
- [指标A]：[精确计算方式]
- [细分X]：[成员资格的精确确定方式]
- [时间段]：[开始日期] 至 [结束日期]，[时区]

### 方法论
1. [分析方法第一步]
2. [第二步]
3. [第三步]

### 假设与局限
- [假设1及其合理性]
- [局限1及其对结论的潜在影响]

### 关键发现
1. [发现1及佐证]
2. [发现2及佐证]

### SQL 查询
[所用全部查询，含注释]

### 注意事项
- [读者在采取行动前应了解的内容]
```

## 提示

- 任何高风险汇报或决策前都运行 `/validate-data`
- 即使是快速分析也能从合理性检查中受益——花一分钟，可以保住你的信誉
- 如果验证发现了问题，修正后再重新验证
- 将验证报告与分析一并分享，有助于建立干系人信任
