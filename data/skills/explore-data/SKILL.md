# /explore-data - 剖析和探索数据集

> 如遇到不熟悉的占位符或需要查看哪些工具已接入，请参阅 [CONNECTORS.md](../../CONNECTORS.md)。

对表或上传的文件生成全面的数据概览。在深入分析前，先了解数据的结构、质量和规律。

## 使用方式

```
/explore-data <表名或文件>
```

## 工作流

### 1. 访问数据

**如果已接入数据仓库 MCP 服务器：**

1. 解析表名（处理 Schema 前缀，如有歧义则给出候选匹配）
2. 查询表元数据：字段名、类型、描述（如有）
3. 对实时数据运行剖析查询

**如果提供了文件（CSV、Excel、Parquet、JSON）：**

1. 读取文件并加载为工作数据集
2. 从数据中推断字段类型

**如果两者都没有：**

1. 请用户提供表名（配合已接入的数据仓库）或上传文件
2. 如果用户描述了表 Schema，提供剖析查询的建议

### 2. 了解结构

分析任何数据前，先了解其结构：

**表级问题：**
- 有多少行和列？
- 数据粒度是什么（每行代表什么）？
- 主键是什么？是否唯一？
- 数据最后更新时间是？
- 数据覆盖多远的历史？

**字段分类** — 将每个字段归类为以下之一：
- **标识符**：唯一键、外键、实体 ID
- **维度**：用于分组/过滤的分类属性（状态、类型、地区、类别）
- **指标**：用于度量的定量值（营收、计数、时长、评分）
- **时间**：日期和时间戳（created_at、updated_at、event_date）
- **文本**：自由格式文本字段（描述、备注、名称）
- **布尔值**：真/假标志
- **结构化**：JSON、数组、嵌套结构

### 3. 生成数据概览

运行以下剖析检查：

**表级指标：**
- 总行数
- 字段数及类型分布
- 大致表大小（如可从元数据获取）
- 日期字段的覆盖范围（日期字段的最小/最大值）

**所有字段：**
- 空值数量和空值率
- 唯一值数量和基数比（唯一值/总行数）
- 最常见的值（前 5-10 个及其频率）
- 最少见的值（后 5 个，用于发现异常）

**数值型字段（指标）：**
```
最小值、最大值、均值、中位数（p50）
标准差
分位数：p1、p5、p25、p75、p95、p99
零值数量
负值数量（如属非预期）
```

**字符串字段（维度、文本）：**
```
最小长度、最大长度、平均长度
空字符串数量
规律分析（值是否遵循某种格式？）
大小写一致性（全大写、全小写、混合？）
前后空格数量
```

**日期/时间戳字段：**
```
最小日期、最大日期
空日期
未来日期（如属非预期）
按月/周的分布
时间序列断层
```

**布尔字段：**
```
真值数量、假值数量、空值数量
真值率
```

**以清晰的汇总表呈现概览**，按字段类型分组（维度、指标、日期、标识符）。

### 4. 识别数据质量问题

应用以下质量评估框架，标记潜在问题：

- **高空值率**：空值率 >5%（警告）、>20%（告警）
- **基数异常低**：应该是高基数的字段却很低（例如，`user_id` 只有 50 个唯一值）
- **基数异常高**：应该是分类字段却有过多唯一值
- **可疑值**：仅预期正值却出现负值、历史数据中出现未来日期、明显的占位符（如 "N/A"、"TBD"、"test"、"999999"）
- **重复检测**：检查是否存在自然键及其是否有重复
- **分布严重倾斜**：极度偏斜的数值分布可能影响均值

### 5. 发现关联和规律

剖析完单个字段后：

- **外键候选**：可能关联其他表的 ID 字段
- **层级结构**：形成自然下钻路径的字段（国家 > 省份 > 城市）
- **相关性**：同向变化的数值字段
- **衍生字段**：看起来由其他字段计算得出的字段
- **冗余字段**：包含相同或近似信息的字段

### 6. 推荐有意义的维度和指标

根据字段概览，推荐：

- **最佳维度字段**，用于数据切分（基数合理的分类字段，3-50 个值）
- **关键指标字段**，用于度量（分布有意义的数值字段）
- **时间字段**，适合趋势分析
- **自然分组**或数据中呈现的层级结构
- **潜在关联键**，链接到其他表（ID 字段、外键）

### 7. 推荐后续分析

建议 3-5 个用户接下来可以执行的具体分析：

- "按 [维度] 对 [指标] 进行 [时间字段] 的趋势分析"
- "对 [偏斜字段] 进行分布深入分析，了解异常值"
- "对 [问题字段] 进行数据质量调查"
- "[指标A] 与 [指标B] 的相关性分析"
- "基于 [日期字段] 和 [状态字段] 的同期群分析"

## 质量评估框架

### 完整性评分

对每个字段评级：
- **完整**（>99% 非空）：绿色
- **基本完整**（95-99%）：黄色 — 调查空值
- **不完整**（80-95%）：橙色 — 了解原因及影响
- **稀疏**（<80%）：红色 — 未经补全可能无法使用

### 一致性检查

关注：
- **值格式不一致**：同一概念的不同表述（"中国"、"CN"、"PRC"）
- **类型不一致**：数字存为字符串、日期格式各异
- **参照完整性**：找不到父记录的外键
- **业务规则违反**：负数量、结束日期早于开始日期、百分比超过 100
- **跨字段一致性**：状态为"已完成"但 completed_at 为空

### 准确性指标

表明准确性问题的危险信号：
- **占位符值**：0、-1、999999、"N/A"、"TBD"、"test"、"xxx"
- **默认值**：某个值的频率异常高
- **过时数据**：活跃系统中 updated_at 长时间未更新
- **不可能的值**：年龄超过 150、遥远的未来日期、负时长
- **整数偏差**：所有值均以 0 或 5 结尾（表明是估算而非测量）

### 时效性评估

- 表最后更新时间是？
- 预期更新频率是多少？
- 事件时间和加载时间之间是否存在延迟？
- 时间序列中是否有断层？

## 规律发现技术

### 分布分析

对数值字段，描述其分布特征：
- **正态**：均值和中位数接近，呈钟形
- **右偏**：高值的长尾（常见于营收、会话时长）
- **左偏**：低值的长尾（较少见）
- **双峰**：两个峰值（表明存在两个不同群体）
- **幂律**：少量极大值，大量小值（常见于用户活跃度）
- **均匀**：各范围内频率大致相等（通常是合成或随机数据）

### 时间规律

对时间序列数据，关注：
- **趋势**：持续的上升或下降走势
- **季节性**：重复出现的规律（每周、每月、每季、每年）
- **星期效应**：工作日与周末的差异
- **节假日效应**：已知节假日前后的下滑或峰值
- **拐点**：水平或趋势的突然变化
- **异常**：打破规律的个别数据点

### 细分发现

通过以下方式识别自然细分：
- 找出有 3-20 个唯一值的分类字段
- 比较不同细分值下的指标分布
- 寻找行为显著不同的细分
- 检验细分是否同质或包含子细分

### 相关性探索

数值字段之间：
- 计算所有指标对的相关矩阵
- 标记强相关（|r| > 0.7）以供调查
- 注意：相关性不等于因果关系——明确指出这一点
- 检查非线性关系（如二次、对数关系）

## Schema 理解与文档

### Schema 文档模板

在为团队使用记录数据集时：

```markdown
## 表：[schema.table_name]

**描述**：[该表代表什么]
**粒度**：[每行代表一个……]
**主键**：[字段名]
**行数**：[大约，含日期]
**更新频率**：[实时 / 每小时 / 每日 / 每周]
**负责人**：[负责的团队或个人]

### 关键字段

| 字段 | 类型 | 描述 | 示例值 | 备注 |
|--------|------|-------------|----------------|-------|
| user_id | STRING | 用户唯一标识 | "usr_abc123" | FK to users.id |
| event_type | STRING | 事件类型 | "click", "view", "purchase" | 15 个唯一值 |
| revenue | DECIMAL | 以美元计的交易金额 | 29.99, 149.00 | 非购买事件为空 |
| created_at | TIMESTAMP | 事件发生时间 | 2024-01-15 14:23:01 | 按该字段分区 |

### 关联关系
- 通过 `user_id` 关联 `users`
- 通过 `product_id` 关联 `products`
- 通过 `event_id` 是 `event_details` 的父表（1:多）

### 已知问题
- [列出已知的数据质量问题]
- [注明分析人员的注意事项]

### 常见查询模式
- [该表的典型使用场景]
```

### Schema 探索查询

连接数据仓库时，使用这些模式来发现 Schema：

```sql
-- 列出 Schema 中的所有表（PostgreSQL）
SELECT table_name, table_type
FROM information_schema.tables
WHERE table_schema = 'public'
ORDER BY table_name;

-- 字段详情（PostgreSQL）
SELECT column_name, data_type, is_nullable, column_default
FROM information_schema.columns
WHERE table_name = 'my_table'
ORDER BY ordinal_position;

-- 表大小（PostgreSQL）
SELECT relname, pg_size_pretty(pg_total_relation_size(relid))
FROM pg_catalog.pg_statio_user_tables
ORDER BY pg_total_relation_size(relid) DESC;

-- 所有表的行数（通用方式）
-- 逐表运行：SELECT COUNT(*) FROM table_name
```

### 数据血缘与依赖

探索陌生数据环境时：

1. 从"输出"表开始（报告或仪表板消费的表）
2. 向上游追溯：哪些表为其提供数据？
3. 识别原始层/暂存层/数据集市层
4. 梳理从原始数据到分析表的转换链
5. 标注数据在哪里被丰富、过滤或聚合

## 提示

- 对于非常大的表（1亿行以上），剖析查询默认使用采样——如需精确计数，请说明
- 第一次探索新数据集时，此命令让你在编写具体查询前先掌握全貌
- 质量标记是启发式的——并非每个标记都是真实问题，但每个都值得快速检查
