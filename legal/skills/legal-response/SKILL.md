# /legal-response — 从模板生成回复

> 如果遇到不熟悉的占位符或需要查看已连接的工具，请参阅 [CONNECTORS.md](../../CONNECTORS.md)。

使用配置的模板生成常见法律查询的回复。以具体细节定制回复，并包含不应使用模板回复情况的上报触发器。

**重要提示**：本命令协助处理法律工作流，但不提供法律建议。生成的回复在发送前，尤其对于受监管的通讯，应由具备资质的法律专业人员审阅。

## 调用方式

```
/legal-response [查询类型]
```

常见查询类型：
- `dsr` 或 `data-subject-request` — 数据主体访问/删除/更正请求
- `hold` 或 `discovery-hold` — 诉讼保全通知
- `vendor` 或 `vendor-question` — 供应商法律问题
- `nda` 或 `nda-request` — 业务团队的 NDA 请求
- `privacy` 或 `privacy-inquiry` — 隐私相关问题
- `subpoena` — 传票或法律程序回应
- `insurance` — 保险理赔通知
- `custom` — 使用自定义模板

如未提供查询类型，询问用户需要什么类型的回复并展示可用类别。

## 工作流程

### 第一步：识别查询类型

接受用户提供的查询类型。如果类型不明确，展示可用类别并请求澄清。

### 第二步：加载模板

在本地设置（例如 `legal.local.md` 或模板目录）中查找模板。

**如果已配置模板：**
- 加载该查询类型的适当模板
- 识别所需变量（收件人姓名、日期、具体细节）

**如果未配置模板：**
- 告知用户未找到该查询类型的模板
- 提供帮助创建模板的选项（参见下方模板创建指南）
- 根据查询类型提供合理的默认回复结构

### 第三步：检查上报触发器

在生成任何回复之前，评估此情况是否具有不应使用模板回复的特征。

#### 普遍上报触发器（适用于所有类别）
- 事项涉及潜在诉讼或监管调查
- 查询来自监管机构、政府机构或执法部门
- 回复可能创建具有约束力的法律承诺或放弃权利
- 事项涉及潜在刑事责任
- 涉及媒体关注或可能引起媒体关注
- 情况前所未有（团队之前未处理过）
- 涉及具有冲突要求的多个司法管辖区
- 事项涉及高管或董事会成员

#### 数据主体请求上报触发器
- 请求涉及未成年人数据，或代表未成年人提出
- 请求来自监管机构（非个人）
- 请求涉及受诉讼保全的数据
- 请求者是当前或前任员工，涉及活跃争议或人事事务
- 请求范围异常宽泛或看起来像钓鱼调查
- 请求涉及在具有独特要求的司法管辖区处理的数据
- 请求涉及特殊类别数据（健康、生物特征、遗传）

#### 诉讼保全上报触发器
- 事项涉及潜在刑事责任
- 保全范围不清晰、有争议或可能过于宽泛
- 某些数据是否在范围内存在疑问
- 同一或相关事项存在先前保全
- 保全可能对持续业务运营产生重大影响
- 保全与监管删除要求冲突
- 保管人反对保全范围

#### 供应商问题上报触发器
- 问题涉及争议或潜在违约
- 供应商威胁诉讼或终止合同
- 问题涉及监管合规（不仅仅是合同条款）
- 回复可能创建具有约束力的承诺或放弃权利
- 回复可能影响正在进行的谈判

#### NDA 请求上报触发器
- 对手方是竞争对手
- NDA 涉及政府保密信息
- 业务背景表明 NDA 用于潜在并购交易
- 请求涉及不寻常的主题（AI 训练数据、生物特征数据等）

#### 传票/法律程序上报触发器
- **始终需要法律顾问审阅**（模板仅作为起点）
- 识别出特权问题
- 涉及第三方数据
- 跨境提交问题
- 时间线不合理

**当检测到上报触发器时：**
1. **停止**：不要生成模板回复
2. **警告**：告知用户检测到上报触发器
3. **解释**：描述检测到的触发器及其重要性
4. **建议**：建议适当的上报路径（高级法律顾问、外部法律顾问、特定团队成员）
5. **提供**：提供一个供法律顾问审阅的草稿（清楚标记为"草稿——仅供法律顾问审阅"）而非最终回复

### 第四步：收集具体细节

提示用户提供定制回复所需的详细信息：

**数据主体请求：**
- 请求者姓名和联系方式
- 请求类型（访问、删除、更正、可携性、退出）
- 涉及哪些数据
- 适用法规（GDPR、CCPA、CPRA、其他）
- 回应截止日期

**诉讼保全：**
- 案件名称和参考编号
- 保管人（谁需要保全）
- 保全范围（日期范围、数据类型、系统）
- 外部法律顾问联系方式
- 生效日期

**供应商问题：**
- 供应商名称
- 参考协议（如适用）
- 正在处理的具体问题
- 相关合同条款

**NDA 请求：**
- 发出请求的业务团队和联系人
- 对手方名称
- NDA 目的
- 双方还是单方
- 任何特殊要求

### 第五步：生成回复

用收集到的细节填充模板。确保回复：
- 使用与情况和关系相符的语气（专业、清晰，商业受众不过于法律化）
- 包含该回复类型所需的所有法律要素
- 引用具体日期、截止日期和义务
- 为收件人提供清晰的后续步骤
- 包含适当的免责声明或注意事项

向用户提交草稿供审阅，再建议发送。

#### 定制指南

**必要定制** — 每个模板回复必须定制以下内容：
- 正确的姓名、日期和参考编号
- 情况的具体事实
- 适用的司法管辖区和法规
- 根据收到查询时间计算的正确回应截止日期
- 适当的签名块和联系信息

**语气调整** — 根据以下情况调整语气：
- **受众**：内部还是外部，业务还是法律，个人还是监管机构
- **关系**：新对手方还是现有合作伙伴还是对抗方
- **敏感性**：常规查询还是有争议的事项还是监管调查
- **紧迫性**：标准时间线还是需要加急回应

**司法管辖区特定调整：**
- 核实引用的法规对请求者所在司法管辖区是正确的
- 调整时限以匹配适用法律
- 包含司法管辖区特定的权利信息
- 使用司法管辖区适当的法律术语

### 第六步：模板创建（如果不存在模板）

如果用户想创建新模板，按照模板创建指南（见下文）进行操作，并提交完成的模板供审阅。建议用户将批准的模板保存到其本地设置中以供将来使用。

## 回复类别

### 1. 数据主体请求（DSR）

**子类别**：
- 收到确认
- 身份核实请求
- 履行回复（访问、删除、更正）
- 附解释的部分拒绝
- 附解释的完全拒绝
- 延期通知

**关键模板元素**：
- 引用适用法规（GDPR、CCPA 等）
- 回应的具体时间线
- 身份核实要求
- 数据主体的权利（包括向监管机构投诉的权利）
- 后续联系信息

**示例模板结构**：
```
Subject: Your Data [Access/Deletion/Correction] Request - Reference {{request_id}}

Dear {{requester_name}},

We have received your request dated {{request_date}} to [access/delete/correct] your personal data under [applicable regulation].

[Acknowledgment / verification request / fulfillment details / denial basis]

We will respond substantively by {{response_deadline}}.

[Contact information]
[Rights information]
```

### 2. 诉讼保全（证据保全通知）

**子类别**：
- 向保管人发出初始保全通知
- 保全提醒/定期重申
- 保全修改（范围变更）
- 保全解除

**关键模板元素**：
- 案件名称和参考编号
- 明确的保全义务
- 保全范围（日期范围、数据类型、系统、通讯类型）
- 禁止证据销毁
- 问题联系人
- 确认要求

**示例模板结构**：
```
Subject: LEGAL HOLD NOTICE - {{matter_name}} - Action Required

PRIVILEGED AND CONFIDENTIAL
ATTORNEY-CLIENT COMMUNICATION

Dear {{custodian_name}},

You are receiving this notice because you may possess documents, communications, or data relevant to the matter referenced above.

PRESERVATION OBLIGATION:
Effective immediately, you must preserve all documents and electronically stored information (ESI) related to:
- Subject matter: {{hold_scope}}
- Date range: {{start_date}} to present
- Document types: {{document_types}}

DO NOT delete, destroy, modify, or discard any potentially relevant materials.

[Specific instructions for systems, email, chat, local files]

Please acknowledge receipt of this notice by {{acknowledgment_deadline}}.

Contact {{legal_contact}} with any questions.
```

### 3. 隐私查询

**子类别**：
- Cookie/追踪查询回复
- 隐私政策问题
- 数据共享实践查询
- 儿童数据查询
- 跨境传输问题

**关键模板元素**：
- 引用组织的隐私通知
- 基于当前实践的具体回答
- 相关隐私文件的链接
- 隐私团队的联系信息

### 4. 供应商法律问题

**子类别**：
- 合同状态查询回复
- 修订请求回复
- 合规认证请求
- 审计请求回复
- 保险证明请求

**关键模板元素**：
- 引用适用协议
- 对供应商问题的具体回复
- 任何必要的注意事项或限制
- 后续步骤和时间线

### 5. NDA 请求

**子类别**：
- 发送组织的标准 NDA 表格
- 接受对手方的 NDA（附修订意见）
- 附解释的 NDA 请求拒绝
- NDA 续签或延期

**关键模板元素**：
- NDA 目的
- 标准条款摘要
- 签署说明
- 时间线预期

### 6. 传票/法律程序

**子类别**：
- 收到确认
- 异议信
- 延期请求
- 合规附函

**关键模板元素**：
- 案件参考和司法管辖区
- 具体异议（如有）
- 保全确认
- 合规时间线
- 特权日志参考（如适用）

**关键说明**：传票回复几乎总是需要个性化的法律顾问审阅。模板仅作为起点框架，而非最终回复。

### 7. 保险通知

**子类别**：
- 初始理赔通知
- 补充信息
- 权利保留回复

**关键模板元素**：
- 保单号和覆盖期
- 事项或事件描述
- 事件时间线
- 请求的覆盖确认

## 模板管理方法论

### 模板组织

模板应按类别组织并在团队的本地设置中维护。每个模板应包含：

1. **类别**：模板处理的查询类型
2. **模板名称**：描述性标识符
3. **使用场景**：此模板适用的情况
4. **上报触发器**：不应使用此模板的情况
5. **必需变量**：每次使用时必须定制的信息
6. **模板正文**：含变量占位符的回复文本
7. **后续行动**：发送回复后的标准步骤
8. **最后审阅日期**：模板上次核实准确性的时间

### 模板生命周期

1. **创建**：根据最佳实践和团队意见起草模板
2. **审阅**：法务团队对模板内容进行审阅和批准
3. **发布**：添加到含元数据的模板库
4. **使用**：使用模板生成回复
5. **反馈**：跟踪使用期间模板被修改的情况，识别改进机会
6. **更新**：当法律、政策或最佳实践改变时修订模板
7. **停用**：归档不再适用的模板

## 模板创建指南

帮助用户创建新模板时：

### 1. 定义使用场景
- 此模板处理什么类型的查询？
- 这种情况多久出现一次？
- 典型受众是谁？
- 典型紧急程度如何？

### 2. 识别必需元素
- 每个回复中必须包含哪些信息？
- 适用哪些监管要求？
- 哪些组织政策规范此类回复？

### 3. 定义变量
- 每次使用时会发生什么变化？（姓名、日期、具体情况）
- 什么保持不变？（法律要求、标准措辞）
- 使用清晰的变量名称：`{{requester_name}}`、`{{response_deadline}}`、`{{matter_reference}}`

### 4. 起草模板
- 使用清晰、专业的语言
- 避免商业受众不必要的法律术语
- 包含所有法律要求的元素
- 为所有变量内容添加占位符
- 如用于邮件，包含主题行模板

### 5. 定义上报触发器
- 哪些情况不应使用此模板？
- 哪些特征表明事项需要个性化处理？
- 要具体：模糊的触发器没有用

### 6. 添加元数据
- 模板名称和类别
- 版本号和最后审阅日期
- 作者和审批者
- 后续行动清单

### 模板格式

```markdown
## Template: {{template_name}}
**Category**: {{category}}
**Version**: {{version}} | **Last Reviewed**: {{date}}
**Approved By**: {{approver}}

### Use When
- [Condition 1]
- [Condition 2]

### Do NOT Use When (Escalation Triggers)
- [Trigger 1]
- [Trigger 2]

### Variables
| Variable | Description | Example |
|---|---|---|
| {{var1}} | [what it is] | [example value] |
| {{var2}} | [what it is] | [example value] |

### Subject Line
[Subject template with {{variables}}]

### Body
[Response body with {{variables}}]

### Follow-Up Actions
1. [Action 1]
2. [Action 2]

### Notes
[Any special instructions for users of this template]
```

## 输出格式

```
## 生成的回复：[查询类型]

**收件人**：[收件人]
**主题**：[主题行]

---

[回复正文]

---

### 上报检查
[确认未检测到上报触发器，或标记的触发器及建议]

### 后续行动
1. [发送后的行动]
2. [需要设置的日历提醒]
3. [跟踪或记录要求]
```

## 说明

- 始终在建议发送前向用户提交草稿回复供审阅
- 如通过 MCP 连接了邮件，提供用回复创建邮件草稿的选项
- 跟踪回应截止日期，并提供设置日历提醒的选项
- 对于受监管的回复（DSR、传票），始终注明适用的截止日期和监管要求
- 模板应是活的文件；当用户修改模板回复时，建议更新，以便模板不断改进
