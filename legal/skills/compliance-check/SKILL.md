# /compliance-check — 合规审查

> 如果遇到不熟悉的占位符或需要查看已连接的工具，请参阅 [CONNECTORS.md](../../CONNECTORS.md)。

对拟议行动、产品功能、营销活动或业务举措进行合规检查。

**重要提示**：本命令协助处理法律工作流，但不提供法律建议。合规评估在被依赖使用前应由具备资质的法律专业人员审阅。监管要求频繁变化；请始终通过权威来源核实当前要求。

## 用法

```
/compliance-check $ARGUMENTS
```

## 我需要您提供的信息

描述您计划做什么。示例：
- "我们想推出一个现金奖励的推荐计划"
- "我们在手机应用中添加生物特征认证"
- "我们需要在美国数据中心处理欧盟客户数据"
- "市场团队想在广告中使用客户证言"

## 隐私法规概述

### GDPR（通用数据保护条例）

**适用范围**：适用于处理欧盟/欧洲经济区个人数据，无论处理组织位于何处。

**企业内部法律团队的关键义务**：
- **合法依据**：识别并记录每项处理活动的合法依据（同意、合同、合法利益、法律义务、重要利益、公共任务）
- **数据主体权利**：在 30 天内（复杂请求可延长 60 天）回应访问、更正、删除、可携性、限制和异议请求
- **数据保护影响评估（DPIA）**：对可能对个人造成高风险的处理活动是必须进行的
- **泄露通知**：在获知个人数据泄露后 72 小时内通知监管机构；如高风险，及时通知受影响个人
- **处理记录**：维护第 30 条处理活动记录
- **国际传输**：确保向欧洲经济区外传输时有适当保障措施（标准合同条款、充分性决定、具有约束力的公司规则）
- **DPO 要求**：如需要，任命数据保护官（公共机构、大规模处理特殊类别、大规模系统性监控）

**常见企业内部法律接触点**：
- 审查供应商 DPA 的 GDPR 合规性
- 就隐私设计要求为产品团队提供建议
- 回应监管机构查询
- 管理跨境数据传输机制
- 审查同意机制和隐私通知

### CCPA / CPRA（加利福尼亚消费者隐私法 / 加利福尼亚隐私权法）

**适用范围**：适用于收集加利福尼亚州居民个人信息且满足收入、数据量或数据销售门槛的企业。

**关键义务**：
- **知情权**：消费者可要求披露所收集、使用和共享的个人信息
- **删除权**：消费者可要求删除其个人信息
- **退出权**：消费者可选择退出个人信息的销售或共享
- **更正权**：消费者可要求更正不准确的个人信息（CPRA 新增）
- **限制使用敏感个人信息的权利**：消费者可将敏感个人信息的使用限制在特定目的（CPRA 新增）
- **非歧视性**：不得歧视行使权利的消费者
- **隐私通知**：必须在收集时或收集前提供隐私通知，描述所收集个人信息的类别和目的
- **服务提供商协议**：与服务提供商的合同必须将个人信息的使用限制在指定的业务目的

**回应时限**：
- 10 个工作日内确认收到
- 45 个日历日内作出实质性回应（经通知可延长 45 天）

### 其他需要监控的关键法规

| 法规 | 司法管辖区 | 主要特点 |
|------|-----------|---------|
| **LGPD**（巴西） | 巴西 | 类似 GDPR；需任命 DPO；国家数据保护局（ANPD）执法 |
| **POPIA**（南非） | 南非 | 信息监管机构监督；需登记处理活动 |
| **PIPEDA**（加拿大） | 加拿大（联邦） | 基于同意的框架；OPC 监督；正在现代化 |
| **PDPA**（新加坡） | 新加坡 | 禁止来电登记；强制泄露通知；PDPC 执法 |
| **隐私法**（澳大利亚） | 澳大利亚 | 澳大利亚隐私原则（APPs）；可通知数据泄露方案 |
| **PIPL**（中国） | 中国 | 严格的跨境传输规则；数据本地化要求；CAC 监督 |
| **英国 GDPR** | 英国 | 脱欧后的英国版本；ICO 监督；类似欧盟 GDPR，含英国特定充分性规定 |

## DPA 审查清单

审查数据处理协议或数据处理附录时，验证以下内容：

### 必要元素（GDPR 第 28 条）

- [ ] **主题和期限**：明确定义的处理范围和期限
- [ ] **性质和目的**：对将发生什么处理及其原因的具体描述
- [ ] **个人数据类型**：正在处理的个人数据类别
- [ ] **数据主体类别**：谁的个人数据正在被处理
- [ ] **控制者的义务和权利**：控制者的指示和监督权

### 处理者义务

- [ ] **仅按书面指示处理**：处理者承诺仅按控制者指示处理（法律要求除外）
- [ ] **保密性**：经授权处理的人员已承诺保密
- [ ] **安全措施**：描述适当的技术和组织措施（参考第 32 条）
- [ ] **次级处理者要求**：
  - [ ] 书面授权要求（一般或特定）
  - [ ] 如为一般授权：变更通知及反对机会
  - [ ] 次级处理者通过书面协议受相同义务约束
  - [ ] 处理者对次级处理者的履行仍负责
- [ ] **数据主体权利协助**：处理者将协助控制者回应数据主体请求
- [ ] **安全和泄露协助**：处理者将协助安全义务、泄露通知、DPIA 和事先咨询
- [ ] **删除或退还**：终止时，删除或退还所有个人数据（由控制者选择），并删除现有副本，除非法律要求保留
- [ ] **审计权**：控制者有权进行审计和检查（或接受第三方审计报告）
- [ ] **泄露通知**：处理者将及时通知控制者个人数据泄露（最好在 24-48 小时内；必须使控制者能满足 72 小时监管截止日期）

### 国际传输

- [ ] **已识别传输机制**：标准合同条款、充分性决定、具有约束力的公司规则或其他有效机制
- [ ] **标准合同条款版本**：如适用，使用当前欧盟标准合同条款（2021 年 6 月版本）
- [ ] **正确模块**：选择适当的标准合同条款模块（C2P、C2C、P2P、P2C）
- [ ] **传输影响评估**：如向无充分性决定的国家传输，已完成评估
- [ ] **补充措施**：技术、组织或合同措施，以解决传输影响评估中发现的缺口
- [ ] **英国附录**：如英国个人数据在范围内，包含英国国际数据传输附录

### 实际考量

- [ ] **责任**：DPA 责任条款与（或不与）主服务协议一致
- [ ] **终止一致性**：DPA 期限与服务协议一致
- [ ] **数据位置**：处理位置已明确且可接受
- [ ] **安全标准**：要求具体的安全标准或认证（SOC 2、ISO 27001 等）
- [ ] **保险**：数据处理活动有充足保险覆盖

### 常见 DPA 问题

| 问题 | 风险 | 标准立场 |
|------|------|---------|
| 无通知的空白次级处理者授权 | 失去对处理链的控制 | 要求有反对权的通知 |
| 泄露通知时限超过 72 小时 | 可能无法及时向监管机构通知 | 要求在 24-48 小时内通知 |
| 无审计权（或仅通过第三方报告行使审计权） | 无法核实合规性 | 接受 SOC 2 Type II + 有因审计权 |
| 数据删除时限未指定 | 数据被无限期保留 | 要求在终止后 30-90 天内删除 |
| 未指定数据处理位置 | 数据可能在任何地方处理 | 要求披露处理位置 |
| 过时的标准合同条款 | 无效的传输机制 | 要求使用当前欧盟标准合同条款（2021 年版本） |

## 数据主体请求处理

### 请求接收

收到数据主体请求时：

1. **识别请求类型**：
   - 访问（个人数据副本）
   - 更正（更正不准确的数据）
   - 删除（"被遗忘权"）
   - 限制处理
   - 数据可携性（结构化、机器可读格式）
   - 反对处理
   - 退出销售/共享（CCPA/CPRA）
   - 限制使用敏感个人信息（CPRA）

2. **识别适用法规**：
   - 数据主体位于哪里？
   - 根据您组织的存在和活动，哪些法律适用？
   - 具体要求和时限是什么？

3. **核实身份**：
   - 确认请求者是其自称的人
   - 使用与数据敏感性相称的合理核实措施
   - 不要求提供过多文件

4. **记录请求**：
   - 收到日期
   - 请求类型
   - 请求者身份
   - 适用法规
   - 回应截止日期
   - 指定处理人

### 回应时限

| 法规 | 初始确认 | 实质性回应 | 延长 |
|------|---------|-----------|------|
| GDPR | 未指定（最佳实践：及时） | 30 天 | +60 天（经通知） |
| CCPA/CPRA | 10 个工作日 | 45 个日历日 | +45 天（经通知） |
| 英国 GDPR | 未指定（最佳实践：及时） | 30 天 | +60 天（经通知） |
| LGPD | 未指定 | 15 天 | 有限延长 |

### 豁免和例外

在履行请求之前，检查是否适用任何豁免：

**各法规的常见豁免**：
- 建立或行使法律主张进行辩护
- 要求保留的法律义务
- 公共利益或官方权力
- 言论自由和信息自由（针对删除请求）
- 公共利益存档或科学/历史研究

**组织特定考量**：
- 诉讼保全：受法律保全的数据不能删除
- 监管保留：财务记录、就业记录和其他类别可能有强制性保留期
- 第三方权利：履行请求可能损害他人权利

### 回应流程

1. 跨系统收集请求者的所有个人数据
2. 应用任何豁免并记录依据
3. 准备回应：履行请求或解释（全部或部分）为何无法履行
4. 如拒绝（全部或部分）：引用拒绝的具体法律依据
5. 告知请求者其向监管机构提出投诉的权利
6. 记录回应并保存请求和回应的记录

## 监管监控基础

### 需要监控的内容

保持对以下发展的了解：
- **监管指导**：监管机构的新指导或更新指导（ICO、CNIL、FTC、州检察长等）
- **执法行动**：表明监管重点的罚款、命令和和解
- **立法变化**：新隐私法、现有法律修订、实施条例
- **行业标准**：ISO 27001、SOC 2、NIST 框架和行业特定要求的更新
- **跨境传输发展**：充分性决定、标准合同条款更新、数据本地化要求

### 监控方法

1. **订阅监管机构通讯**（简报、RSS 订阅、官方公告）
2. **跟踪相关法律出版物**，了解新发展的分析
3. **查阅行业协会更新**，获取行业特定指导
4. **维护监管日历**，记录已知的即将到来的截止日期、生效日期和合规里程碑
5. **向法务团队简报**，说明影响组织处理活动的重要发展

### 上报标准

在以下情况下将监管发展上报给高级法律顾问或领导层：
- 新法规或指导直接影响组织的核心业务活动
- 组织所在行业的执法行动表明监管审查力度加强
- 需要组织变革的合规截止日期临近
- 组织依赖的数据传输机制受到挑战或被认定无效
- 监管机构就涉及组织的事项发起查询或调查

## 提示

1. **要具体** — "我们想给所有用户发邮件"比"营销活动"更好。
2. **包含地理范围** — 合规要求因司法管辖区而异。
3. **提及数据** — 涉及哪些个人数据？这是大多数合规要求的驱动因素。
