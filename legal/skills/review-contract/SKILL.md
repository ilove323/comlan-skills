# /review-contract — 对照规范手册进行合同审查

> 如果遇到不熟悉的占位符或需要查看已连接的工具，请参阅 [CONNECTORS.md](../../CONNECTORS.md)。

对照您组织的谈判规范手册审查合同。分析每个条款，标记偏差，生成修订建议，并提供商业影响分析。

**重要提示**：AI助手协助处理法律工作流，但不提供法律建议。所有分析在被依赖使用前应由具备资质的法律专业人员审阅。

## 调用方式

```
/review-contract <合同文件或 URL>
```

审查合同：@$1

## 工作流程

### 第一步：接受合同

以以下任何格式接受合同：
- **文件上传**：PDF、DOCX 或其他文档格式
- **URL**：CLM、云存储（例如 Box、Egnyte、SharePoint）或其他文档系统中的合同链接
- **粘贴文本**：直接粘贴到对话中的合同文本

如果未提供合同，提示用户提供。

### 第二步：收集背景

开始审查前询问用户背景：

1. **您代表哪方？**（供应商/供应商方、客户/购买方、许可方、被许可方、合作伙伴——或其他）
2. **截止日期**：何时需要完成？（影响问题的优先级排序）
3. **重点领域**：有任何具体关注点吗？（例如，"数据保护至关重要"、"我们需要期限上的灵活性"、"知识产权归属是关键问题"）
4. **交易背景**：有什么相关业务背景吗？（例如，交易规模、战略重要性、现有关系）

如果用户提供部分背景，用现有信息继续并注明假设。

### 第三步：加载规范手册

在本地设置（例如 `legal.local.md` 或类似配置文件）中查找组织的合同审查规范手册。

规范手册应定义：
- **标准立场**：组织对每种主要条款类型的首选条款
- **可接受范围**：无需上报即可同意的条款
- **上报触发器**：需要高级法律顾问审查或外部法律顾问介入的条款

**如果未配置规范手册：**
- 告知用户未找到规范手册
- 提供两个选项：
  1. 帮助用户建立规范手册（引导完成定义主要条款立场的过程）
  2. 使用通用的广泛接受的商业标准作为基准继续进行通用审查
- 如果进行通用审查，明确说明审查基于一般商业标准而非组织的具体立场

### 第四步：逐条分析

应用以下审查流程：

1. **识别合同类型**：SaaS 协议、专业服务、许可证、合作伙伴关系、采购等。合同类型影响哪些条款最重要。
2. **确定用户方**：供应商、客户、许可方、被许可方、合作伙伴。这从根本上改变分析（例如，责任限制保护偏向不同方）。
3. **标记问题之前阅读完整合同**。条款相互影响（例如，无上限赔偿可能部分被宽泛的责任限制所缓解）。
4. **对照规范手册立场分析每个实质性条款**。
5. **整体考虑合同**：整体风险分配和商业条款是否平衡？

系统地分析合同，至少涵盖：

| 条款类别 | 关键审查要点 |
|---------|-----------|
| **责任限制** | 上限金额、豁免、双方还是单方、间接损害 |
| **赔偿** | 范围、双方还是单方、上限、知识产权侵权、数据泄露 |
| **知识产权归属** | 既有知识产权、开发的知识产权、雇佣作品、许可授予、转让 |
| **数据保护** | DPA 要求、处理条款、次级处理方、泄露通知、跨境传输 |
| **保密性** | 范围、期限、豁免、归还/销毁义务 |
| **陈述与保证** | 范围、免责声明、存续期 |
| **期限与终止** | 期限、续期、便利终止、有因终止、过渡期 |
| **适用法律与争议解决** | 司法管辖区、场所、仲裁与诉讼 |
| **保险** | 覆盖要求、最低限额、覆盖证明 |
| **转让** | 同意要求、控制权变更、例外情况 |
| **不可抗力** | 范围、通知、终止权利 |
| **付款条款** | 账期、滞纳金、税款、价格上调 |

对每个条款，对照规范手册（或通用标准）进行评估，并注明其是否存在、缺失或异常。

#### 详细条款指南

##### 责任限制

**需要审查的关键要素：**
- 上限金额（固定金额、费用的倍数或无上限）
- 上限是否相互适用或对各方不同
- 上限的豁免（哪些责任无上限）
- 是否排除间接、特殊或惩罚性损害
- 排除是否相互适用
- 间接损害排除的豁免
- 上限是按每次索赔、每年还是累计适用

**常见问题：**
- 上限设定为已支付费用的一小部分（例如，低价值合同的"过去 3 个月已支付费用"）
- 有利于起草方的不对称豁免
- 实际上消除上限的宽泛豁免（例如，"违反 X 节的任何行为"，而 X 节涵盖大多数义务）
- 一方违约没有间接损害排除

##### 赔偿

**需要审查的关键要素：**
- 赔偿是双方还是单方的
- 范围：什么触发赔偿义务（知识产权侵权、数据泄露、人身伤害、陈述与保证违约）
- 赔偿是否有上限（通常受整体责任上限约束，有时无上限）
- 程序：通知要求、控制辩护的权利、和解的权利
- 被赔偿方是否必须减轻损失
- 赔偿与责任限制条款之间的关系

**常见问题：**
- 双方均贡献知识产权时单方知识产权侵权赔偿
- "任何违约"的赔偿（过于宽泛；实际上将责任上限转变为无上限责任）
- 无权控制索赔的辩护
- 终止后无限期存续的赔偿义务

##### 知识产权

**需要审查的关键要素：**
- 既有知识产权的归属（各方应保留自己的知识产权）
- 合作期间开发的知识产权的归属
- 雇佣作品条款及其范围
- 许可授予：范围、独占性、领域、再许可权利
- 开源考量
- 反馈条款（对建议或改进的授予）

**常见问题：**
- 可能捕获客户既有知识产权的宽泛知识产权转让
- 超出可交付成果范围的雇佣作品条款
- 授予永久、不可撤销许可的无限制反馈条款
- 超出业务关系所需的许可范围

##### 数据保护

**需要审查的关键要素：**
- 是否需要数据处理协议/附录（DPA）
- 数据控制者与数据处理者的分类
- 次级处理者权利和通知义务
- 数据泄露通知时限（GDPR 的 72 小时）
- 跨境数据传输机制（标准合同条款、充分性决定、具有约束力的公司规则）
- 终止时的数据删除或归还义务
- 数据安全要求和审计权
- 数据处理的目的限制

**常见问题：**
- 处理个人数据时没有 DPA
- 无通知的次级处理者空白授权
- 超过监管要求的泄露通知时限
- 数据国际传输时没有跨境传输保护
- 数据删除条款不充分

##### 期限和终止

**需要审查的关键要素：**
- 初始期限和续期期限
- 自动续期条款和通知期
- 便利终止：是否可用？通知期？提前终止费？
- 有因终止：治愈期？什么构成有因？
- 终止的影响：数据归还、过渡协助、存续条款
- 过渡期和义务

**常见问题：**
- 没有便利终止的长期限
- 通知窗口短的自动续期（例如年度续期的 30 天通知）
- 有因终止没有治愈期
- 过渡协助条款不充分
- 实际上无限期延长协议的存续条款

##### 适用法律和争议解决

**需要审查的关键要素：**
- 法律选择（适用司法管辖区）
- 争议解决机制（诉讼、仲裁、先调解）
- 诉讼的场所和司法管辖区
- 仲裁规则和仲裁地（如仲裁）
- 陪审团放弃
- 集体诉讼放弃
- 胜诉方律师费

**常见问题：**
- 不利的司法管辖区（异常或偏远的场所）
- 有利于起草方规则的强制仲裁
- 无相应保护的陪审团审判放弃
- 正式争议解决前没有升级程序

### 第五步：标记偏差

使用三级系统对规范手册的每个偏差进行分类：

#### 绿色 — 可接受

条款与组织的标准立场一致或优于标准立场。不会实质性增加风险的轻微变化，在商业上是合理的。

**示例：**
- 标准为 12 个月时，责任上限为 18 个月费用（对客户更有利）
- 标准为 3 年时，双方 NDA 期限为 2 年（较短但合理）
- 接近首选司法管辖区的成熟商业司法管辖区的适用法律

**行动**：注意以供了解。无需谈判。

#### 黄色 — 谈判

条款超出标准立场但在可谈判范围内。该条款在市场上常见但不是组织的首选。需要关注并可能需要谈判，但不需要上报。

**示例：**
- 标准为 12 个月时，责任上限为 6 个月费用（低于标准但可谈判）
- 标准为双方时，单方知识产权侵权赔偿（常见市场立场但非首选）
- 标准为 90 天时，自动续期为 60 天通知
- 在可接受但非首选的司法管辖区的适用法律

**行动**：生成具体的修订措辞。提供退守立场。估计接受与谈判的商业影响。
- **包含**：将条款带回标准立场的具体修订措辞
- **包含**：对手方反对时的退守立场
- **包含**：按现状接受与谈判的商业影响

#### 红色 — 上报

条款超出可接受范围，触发了定义的上报标准，或构成重大风险。需要高级法律顾问审查、外部法律顾问介入或业务决策者签字。

**示例：**
- 无上限责任或没有责任限制条款
- 无上限的单方宽泛赔偿
- 既有知识产权的转让
- 处理个人数据时未提供 DPA
- 不合理的非竞争或排他性条款
- 有强制仲裁的不利司法管辖区的适用法律

**行动**：解释具体风险。提供市场标准的替代措辞。估计敞口。建议上报路径。
- **包含**：为什么这是红色标志（具体风险）
- **包含**：标准市场立场是什么样的
- **包含**：商业影响和潜在敞口
- **包含**：建议的上报路径

### 第六步：生成修订建议

对每个黄色和红色偏差，提供：
- **当前措辞**：引用相关合同文本
- **建议修订**：具体的替代措辞
- **理由**：适合与对手方共享的简短解释
- **优先级**：谈判中这是必须有的还是可有可无的

#### 修订生成最佳实践

生成修订建议时：

1. **要具体**：提供准确的措辞，而不是模糊的指导。修订应可直接插入。
2. **要平衡**：提出在关键点上坚定但在商业上合理的措辞。过于激进的修订会拖慢谈判。
3. **解释理由**：包含适合与对手方法律顾问共享的简短、专业的理由。
4. **提供退守立场**：对于黄色事项，包含如果主要诉求被拒绝的退守立场。
5. **确定优先级**：不是所有修订都同等重要。说明哪些是必须有的，哪些是可有可无的。
6. **考虑关系**：根据这是新供应商、战略合作伙伴还是商品供应商来调整语气和方法。

#### 修订格式

对每个修订：
```
**条款**：[节参考和条款名称]
**当前措辞**："[合同中的准确引用]"
**建议修订**："[具体替代措辞，概念上以粗体表示添加，以删除线表示删除]"
**理由**：[解释原因的 1-2 句话，适合对外共享]
**优先级**：[必须有 / 应该有 / 可有可无]
**退守立场**：[如果主要修订被拒绝时的替代立场]
```

### 第七步：商业影响摘要

提供一个摘要节，涵盖：
- **整体风险评估**：合同风险状况的高层次视角
- **前 3 个问题**：需要解决的最重要事项
- **谈判策略**：建议的方法（哪些问题先提，什么可以让步）
- **时间线考量**：影响谈判方法的任何紧迫因素

#### 谈判优先级框架

呈现修订时，按谈判优先级组织：

**第一层 — 必须有（交易破坏者）**
组织无法在未解决情况下推进的问题：
- 无上限或实质不充分的责任保护
- 受监管数据缺少数据保护要求
- 可能危及核心资产的知识产权条款
- 与监管义务冲突的条款

**第二层 — 应该有（强烈偏好）**
实质影响风险但有谈判空间的问题：
- 在范围内的责任上限调整
- 赔偿范围和双方性
- 终止灵活性
- 审计和合规权利

**第三层 — 可有可无（让步候选）**
改善立场但可以战略性让步的问题：
- 首选适用法律（如果替代方案可接受）
- 通知期偏好
- 轻微的定义改进
- 保险证明要求

**谈判策略**：以第一层事项开始。用第三层让步换取第二层胜利。不要在没有上报的情况下在第一层让步。

### 第八步：CLM 路由（如已连接）

如果通过 MCP 连接了合同生命周期管理系统：
- 根据合同类型和风险级别推荐适当的审批工作流
- 建议正确的路由路径（例如，标准审批、高级法律顾问、外部法律顾问）
- 注明基于合同价值或风险标志所需的任何审批

如果未连接 CLM，跳过此步骤。

## 输出格式

按以下结构输出：

```
## 合同审查摘要

**文件**：[合同名称/标识符]
**各方**：[各方名称和角色]
**您的立场**：[供应商/客户等]
**截止日期**：[如提供]
**审查依据**：[规范手册 / 通用标准]

## 关键发现

[前 3-5 个问题，含严重程度标志]

## 逐条分析

### [条款类别] — [绿色/黄色/红色]
**合同规定**：[条款摘要]
**规范手册立场**：[您的标准]
**偏差**：[差距描述]
**商业影响**：[这在实际中意味着什么]
**修订建议**：[具体措辞，如果是黄色或红色]

[对每个主要条款重复]

## 谈判策略

[建议方法、优先级、让步候选]

## 后续步骤

[具体行动]
```

## 说明

- 如果合同是英文以外的语言，注明此情况并询问用户是否需要翻译或以原语言进行审查
- 对于非常长的合同（50 页以上），提供先聚焦于最重要节，再进行完整审查的选项
- 始终提醒用户，此分析在被用于法律决策前应由具备资质的法律顾问审阅
