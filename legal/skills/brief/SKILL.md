# /brief — 法务团队简报

> 如果遇到不熟悉的占位符或需要查看已连接的工具，请参阅 [CONNECTORS.md](../../CONNECTORS.md)。

为法务工作生成上下文简报。支持三种模式：每日简报、专题简报和事件简报。

**重要提示**：本命令协助处理法律工作流，但不提供法律建议。简报在被依赖使用前应由具备资质的法律专业人员审阅。

## 调用方式

```
/brief daily              # 法律相关事项的早间简报
/brief topic [查询]       # 特定法律问题的研究简报
/brief incident [主题]    # 突发情况的快速简报
```

如未指定模式，询问用户需要哪种类型的简报。

## 模式

---

### 每日简报

法务团队成员开始一天工作所需了解的所有事项的早间摘要。

#### 扫描来源

检查每个已连接来源中的法律相关事项：

**邮件（如已连接）：**
- 新的合同请求或审查请求
- 合规问题或报告
- 对手方就进行中谈判的回复
- 法务团队收件箱中的标记或紧急事项
- 外部法律顾问的往来邮件
- 监管或法律更新简报

**日历（如已连接）：**
- 今日需要法律准备的会议（董事会会议、交易审查、供应商电话）
- 本周即将到来的截止日期（合同到期、归档截止、回复截止）
- 定期法务团队同步会议

**即时通讯（如已连接）：**
- 法务团队频道的隔夜消息
- 请求法律意见的私信
- 法律相关话题的提及（合同、合规、隐私、NDA、条款）
- 升级或紧急请求

**CLM（如已连接）：**
- 等待审查或签署的合同
- 即将到期日期（未来 30 天内）
- 新执行的协议

**CRM（如已连接）：**
- 进入需要法律介入阶段的交易
- 标记为需要法律审查的新商机

#### 输出格式

```
## 每日法律简报 — [日期]

### 紧急 / 需要行动
[需要立即关注的事项，按紧急程度排序]

### 合同管道
- **等待您审查**：[数量和列表]
- **等待对手方回复**：[数量和列表]
- **即将到期**：[本周到期的事项]

### 新请求
[自上次简报以来收到的合同审查请求、NDA 请求、合规问题]

### 今日日历
[具有法律相关性的会议及所需准备]

### 团队动态
[法务团队频道的关键消息或更新]

### 本周截止日期
[即将到来的截止日期和归档日期]

### 不可用来源
[任何未连接或返回错误的来源]
```

---

### 专题简报

就特定法律问题或主题跨可用来源进行研究和简报。

#### 工作流程

1. 接受用户提供的主题查询
2. 跨已连接来源搜索：
   - **文件**：内部备忘录、先前分析、规范手册、先例
   - **邮件**：该主题的先前往来邮件
   - **即时通讯**：团队关于该主题的讨论
   - **CLM**：相关合同或条款
3. 将发现综合为结构化简报

#### 输出格式

```
## 专题简报：[主题]

### 摘要
[发现的 2-3 句执行摘要]

### 背景
[来自内部来源的背景和历史]

### 当前状态
[基于可用文件，组织当前的立场或方法]

### 关键考量
[重要因素、风险或待解问题]

### 内部先例
[在内部来源中找到的先前决策、备忘录或立场]

### 缺口
[缺少哪些信息，哪些来源不可用]

### 建议的后续步骤
[用户应如何处理这些信息]
```

#### 重要说明
- 专题简报综合已连接来源中的可用内容；不能替代正式法律研究
- 如果主题需要当前法律依据或判例法，建议用户咨询法律研究平台（Westlaw、Lexis 等）或外部法律顾问
- 始终注明所搜索来源的局限性

---

### 事件简报

针对需要立即法律关注的突发情况的快速简报（数据泄露、诉讼威胁、监管查询、知识产权争议等）。

#### 工作流程

1. 接受事件主题或描述
2. 快速扫描所有已连接来源以获取相关背景：
   - **邮件**：关于该事件的往来邮件
   - **即时通讯**：实时讨论和升级
   - **文件**：相关政策、应对计划、保险覆盖
   - **日历**：已安排的应对会议
   - **CLM**：受影响的合同、赔偿条款、保险要求
3. 汇编为可行动的事件简报

#### 输出格式

```
## 事件简报：[主题]
**准备时间**：[时间戳]
**分类**：[如可确定，填写严重程度评估]

### 情况摘要
[关于事件已知信息]

### 时间线
[基于可用来源的事件按时间顺序摘要]

### 即时法律考量
[监管通知要求、保全义务、特权考量]

### 相关协议
[可能涉及的合同、保险单或其他协议]

### 内部应对
[基于邮件/即时通讯，已发生的应对活动]

### 关键联系人
[从来源中识别的相关内外部联系人]

### 建议的即时行动
1. [最紧急的行动]
2. [第二优先]
3. [以此类推]

### 信息缺口
[尚不清楚且需要确认的内容]

### 已检查的来源
[搜索了什么，哪些不可用]
```

#### 事件简报的重要说明
- 速度至关重要。用可用信息快速生成简报，而不是等待完整信息
- 如有诉讼保全或保存义务，立即标记
- 注意特权考量（如适用，将简报标记为律师-委托人特权/工作成果保护）
- 如果事件可能涉及数据泄露，标记适用的通知截止日期（例如 GDPR 的 72 小时）
- 如果事件重大，建议聘请外部法律顾问

## 一般说明

- 如果来源不可用，在简报中突出说明缺口，让用户知道哪些内容未被检查
- 对于每日简报，随时间了解用户偏好（他们觉得有用的内容，他们希望过滤掉的内容）
- 简报应具有可操作性：每个事项都应有明确的后续步骤或纳入原因
- 保持简报简洁。链接到源材料，而不是完整复制
