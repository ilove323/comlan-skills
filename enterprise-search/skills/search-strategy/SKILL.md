# 搜索策略

> 如果您看到不熟悉的占位符或需要检查已连接的工具，请参阅 [CONNECTORS.md](../../CONNECTORS.md)。

企业搜索背后的核心智能。将单个自然语言问题转化为并行、针对来源的搜索，并生成排序、去重的结果。

## 目标

将这个：
```
"我们就 API 迁移时间表做了什么决定？"
```

转化为跨每个已连接来源的针对性搜索：
```
~~chat:  "API 迁移时间表决定"（语义）+ "API migration" in:#engineering after:2025-01-01
~~knowledge base: 语义搜索"API 迁移时间表决定"
~~project tracker:  相关工作区中的文本搜索"API migration"
```

然后将结果合成为单一连贯的答案。

## 查询分解

### 步骤 1：识别查询类型

对用户的问题进行分类以确定搜索策略：

| 查询类型 | 示例 | 策略 |
|-----------|---------|----------|
| **决定** | "我们就 X 做了什么决定？" | 优先对话（~~chat、邮件），查找结论信号 |
| **状态** | "项目 Y 的状态是什么？" | 优先最近活动、任务追踪器、状态更新 |
| **文档** | "Z 的规格在哪里？" | 优先 Drive、Wiki、共享文档 |
| **人员** | "谁在做 X？" | 搜索任务分配、消息作者、文档协作者 |
| **事实** | "我们关于 X 的政策是什么？" | 优先 Wiki、官方文档，然后确认性对话 |
| **时间** | "X 是什么时候发生的？" | 宽时间范围搜索，查找时间戳 |
| **探索** | "我们对 X 了解什么？" | 跨所有来源的广泛搜索，合成 |

### 步骤 2：提取搜索组件

从查询中提取：

- **关键词**：必须出现在结果中的核心术语
- **实体**：人员、项目、团队、工具（如果可用，使用记忆系统）
- **意图信号**：决定词、状态词、时间标记
- **约束**：时间范围、来源提示、作者过滤器
- **否定**：要排除的内容

### 步骤 3：为每个来源生成子查询

对于每个可用来源，创建一个或多个针对性查询：

**对以下情况优先使用语义搜索：**
- 概念性问题（"我们对...有什么看法"）
- 确切关键词未知的问题
- 探索性查询

**对以下情况优先使用关键词搜索：**
- 已知术语、项目名称、缩写
- 用户引用的确切短语
- 过滤器重的查询（from:、in:、after:）

**当主题可能有不同称呼时，生成多个查询变体：**
```
用户："Kubernetes 设置"
查询："Kubernetes"、"k8s"、"cluster"、"container orchestration"
```

## 针对来源的查询转换

### ~~chat

**语义搜索**（自然语言问题）：
```
query: "Aurora 项目的状态是什么？"
```

**关键词搜索：**
```
query: "project aurora status update"
query: "aurora in:#engineering after:2025-01-15"
query: "from:<@UserID> aurora"
```

**过滤器映射：**
| 企业过滤器 | ~~chat 语法 |
|------------------|--------------|
| `from:sarah` | `from:sarah` 或 `from:<@USERID>` |
| `in:engineering` | `in:engineering` |
| `after:2025-01-01` | `after:2025-01-01` |
| `before:2025-02-01` | `before:2025-02-01` |
| `type:thread` | `is:thread` |
| `type:file` | `has:file` |

### ~~knowledge base（Wiki）

**语义搜索** — 用于概念性查询：
```
descriptive_query: "API 迁移时间表和决定理由"
```

**关键词搜索** — 用于确切术语：
```
query: "API migration"
query: "\"API migration timeline\""  (确切短语)
```

### ~~project tracker

**任务搜索：**
```
text: "API migration"
workspace: [workspace_id]
completed: false  （状态查询）
assignee_any: "me"  （"我的任务"查询）
```

**过滤器映射：**
| 企业过滤器 | ~~project tracker 参数 |
|------------------|----------------|
| `from:sarah` | `assignee_any` 或 `created_by_any` |
| `after:2025-01-01` | `modified_on_after: "2025-01-01"` |
| `type:milestone` | `resource_subtype: "milestone"` |

## 结果排序

### 相关性评分

按以下因素对每条结果评分（按查询类型加权）：

| 因素 | 权重（决定） | 权重（状态） | 权重（文档） | 权重（事实） |
|--------|-------------------|------------------|--------------------|-------------------|
| 关键词匹配 | 0.3 | 0.2 | 0.4 | 0.3 |
| 时效性 | 0.3 | 0.4 | 0.2 | 0.1 |
| 权威性 | 0.2 | 0.1 | 0.3 | 0.4 |
| 完整性 | 0.2 | 0.3 | 0.1 | 0.2 |

### 权威层次

取决于查询类型：

**对于事实/政策问题：**
```
Wiki/官方文档 > 共享文档 > 邮件公告 > 聊天消息
```

**对于"发生了什么"/决定问题：**
```
会议纪要 > 讨论串结论 > 邮件确认 > 聊天消息
```

**对于状态问题：**
```
任务追踪器 > 最近聊天 > 状态文档 > 邮件更新
```

## 处理歧义

当查询有歧义时，提出一个有针对性的澄清问题，而非猜测：

```
歧义："搜索迁移"
→ "我发现了几个迁移的引用。您在找：
   1. 数据库迁移（Project Phoenix）
   2. 云迁移（AWS → GCP）
   3. 邮件迁移（Exchange → O365）"
```

只在以下情况下请求澄清：
- 确实存在会产生非常不同结果的不同解释
- 歧义会显著影响搜索哪些来源

以下情况不请求澄清：
- 查询足够清晰以产生有用结果
- 轻微歧义可以通过返回多种解释的结果来解决

## 降级策略

当来源不可用或无返回结果时：

1. **来源不可用**：跳过它，搜索剩余来源，记录缺口
2. **来源无结果**：尝试更宽泛的查询术语，删除日期过滤器，尝试替代关键词
3. **所有来源无返回**：向用户建议查询修改
4. **速率限制**：记录限制，从其他来源返回结果，建议稍后重试

### 查询扩展

如果初始查询返回结果太少：
```
原始："PostgreSQL migration Q2 timeline decision"
更宽："PostgreSQL migration"
更宽："database migration"
最宽："migration"
```

按以下顺序删除约束：
1. 日期过滤器（搜索所有时间）
2. 来源/位置过滤器
3. 不太重要的关键词
4. 只保留核心实体/主题术语

## 并行执行

始终并行在各来源执行搜索，绝不顺序执行。总搜索时间应大约等于最慢单一来源的时间，而非所有来源的总和。

```
[用户查询]
     ↓ 分解
[~~chat 查询] [~~email 查询] [~~cloud storage 查询] [Wiki 查询] [~~project tracker 查询]
     ↓            ↓            ↓              ↓            ↓
  (并行执行)
     ↓
[合并 + 排序 + 去重]
     ↓
[合成答案]
```
