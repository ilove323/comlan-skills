# 摘要命令

> 如果您看到不熟悉的占位符或需要检查已连接的工具，请参阅 [CONNECTORS.md](../../CONNECTORS.md)。

扫描所有已连接来源的最近活动，生成突出重要内容的结构化摘要。

## 操作说明

### 1. 解析参数

根据用户输入确定时间窗口：

- `--daily` — 过去 24 小时（如未指定参数，默认使用此项）
- `--weekly` — 过去 7 天

用户也可以指定自定义范围：
- `--since yesterday`
- `--since Monday`
- `--since 2025-01-20`

### 2. 检查可用来源

确定哪些 MCP 来源已连接（与搜索命令相同的方法）：

- **~~chat** — 频道、私信、提及
- **~~email** — 收件箱、已发送、讨论串
- **~~cloud storage** — 最近修改的与用户共享的文档
- **~~project tracker** — 分配、完成、评论的任务
- **~~CRM** — 商机更新、客户活动
- **~~knowledge base** — 最近更新的 Wiki 页面

如果没有连接来源，指导用户：
```
要生成摘要，您需要至少连接一个来源。
检查您的 MCP 设置以添加 ~~chat、~~email、~~cloud storage 或其他工具。
```

### 3. 从每个来源收集活动

**~~chat：**
- 搜索提及用户的消息（`to:me`）
- 检查用户所在频道的最近活动
- 查看用户参与的讨论串
- 识别关键频道中的新消息

**~~email：**
- 搜索最近的收件箱消息
- 识别有新回复的讨论串
- 标记包含针对用户的行动项或问题的邮件

**~~cloud storage：**
- 查找最近修改或与用户共享的文档
- 记录用户拥有或协作的文档上的新评论

**~~project tracker：**
- 分配给用户的任务（新建或更新）
- 用户关注的其他人完成的任务
- 用户参与任务上的评论

**~~CRM：**
- 商机阶段变化
- 用户负责账户上记录的新活动
- 更新的联系人或账户

**~~knowledge base：**
- 相关集合中最近更新的文档
- 被关注区域中新创建的文档

### 4. 识别关键内容

从所有收集的活动中，提取和分类：

**行动项：**
- 对用户提出的直接请求（"能否..."、"请..."、"@用户"）
- 分配或即将到期的任务
- 等待用户回复的问题
- 审查请求

**决定：**
- 在讨论串或邮件中达成的结论
- 审批或拒绝
- 政策或方向变化

**提及：**
- 用户被提及或引用的次数
- 关于用户项目或领域的讨论

**更新：**
- 用户关注的项目状态变化
- 用户领域的文档更新
- 用户等待的已完成事项

### 5. 按主题分组

按主题、项目或主线而非按来源组织摘要。合并跨来源的相关活动：

```
## Aurora 项目
- ~~chat: 设计评审讨论串结束——团队选择了方案 B（#design，周二）
- ~~email: Sarah 发送了整合反馈后的更新规格（周三）
- ~~cloud storage: "Aurora API Spec v3" 由 Sarah 更新（周三）
- ~~project tracker: 3 个任务移至进行中，2 个已完成

## 预算规划
- ~~email: 财务团队请求在周五前提交 Q2 预测
- ~~chat: Todd 在 #finance 分享了模板（周一）
- ~~cloud storage: "Q2 预算模板" 与您共享（周一）
```

### 6. 格式化摘要

清晰地构建输出：

```
# [每日/每周] 摘要 — [日期或日期范围]

已扫描来源：~~chat、~~email、~~cloud storage、[其他]

## 行动项（X 个）
- [ ] [行动项 1] — 来自 [人员]，[来源]（[日期]）
- [ ] [行动项 2] — 来自 [人员]，[来源]（[日期]）

## 已做决定
- [决定 1] — [背景]（[来源]，[日期]）
- [决定 2] — [背景]（[来源]，[日期]）

## [主题/项目组 1]
[附有来源归属的活动摘要]

## [主题/项目组 2]
[附有来源归属的活动摘要]

## 提及
- [提及背景] — [来源]（[日期]）

## 已更新文档
- [文档名称] — [谁修改了、改了什么]（[日期]）
```

### 7. 处理不可用来源

如果任何来源失败或不可达：
```
注意：无法访问 [来源名称] 用于此摘要。
以下来源已包含在内：[成功来源列表]。
```

不要让一个失败的来源阻止摘要生成。从可用来源生成最佳摘要。

### 8. 汇总统计

结尾附上快速汇总：
```
---
[X] 个行动项 · [Y] 个决定 · [Z] 次提及 · [W] 次文档更新
涵盖 [N] 个来源 · 覆盖 [时间范围]
```

## 说明

- 如果未指定参数，默认使用 `--daily`
- 按主题/项目分组，而非按来源——用户关心发生了什么，而非在哪里发生的
- 行动项应始终排在首位——它们是摘要中最可操作的部分
- 去除跨来源的重复活动（~~chat 和邮件中的同一决定 = 一个条目）
- 对于每周摘要，重要性优先于完整性——突出重要内容，跳过噪音
- 如果用户有记忆系统（CLAUDE.md），用它来解码人名和项目引用
- 每个条目包含足够背景，使用户无需点击查看详情即可决定是否深入
