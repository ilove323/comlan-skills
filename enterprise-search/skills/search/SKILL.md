# 搜索命令

> 如果您看到不熟悉的占位符或需要检查已连接的工具，请参阅 [CONNECTORS.md](../../CONNECTORS.md)。

在单条查询中搜索所有已连接的 MCP 来源。分解用户的问题，运行并行搜索，并合成结果。

## 操作说明

### 1. 检查可用来源

搜索前，确定哪些 MCP 来源可用。尝试从可用工具列表中识别已连接的工具。常见来源：

- **~~chat** — 聊天平台工具
- **~~email** — 邮件工具
- **~~cloud storage** — 云存储工具
- **~~project tracker** — 项目追踪工具
- **~~CRM** — CRM 工具
- **~~knowledge base** — 知识库工具

如果没有连接 MCP 来源：
```
要跨工具搜索，您需要至少连接一个来源。
检查您的 MCP 设置以添加 ~~chat、~~email、~~cloud storage 或其他工具。

支持的来源：~~chat、~~email、~~cloud storage、~~project tracker、~~CRM、~~knowledge base，
以及任何其他 MCP 连接的服务。
```

### 2. 解析用户查询

分析搜索查询以理解：

- **意图**：用户在找什么？（决定、文档、人员、状态更新、对话）
- **实体**：提到的人员、项目、团队、工具
- **时间约束**：时效性信号（"本周"、"上个月"、具体日期）
- **来源提示**：对特定工具的引用（"在 ~~chat 中"、"那封邮件"、"那个文档"）
- **过滤器**：从查询中提取明确过滤器：
  - `from:` — 按发件人/作者过滤
  - `in:` — 按频道、文件夹或位置过滤
  - `after:` — 只返回此日期之后的结果
  - `before:` — 只返回此日期之前的结果
  - `type:` — 按内容类型过滤（消息、邮件、文档、讨论串、文件）

### 3. 分解为子查询

对于每个可用来源，使用该来源的原生搜索语法创建针对性子查询：

**~~chat：**
- 使用聊天平台的可用搜索和读取工具
- 转换过滤器：`from:` 映射到发件人，`in:` 映射到频道/房间，日期映射到时间范围过滤器
- 适当时使用自然语言查询进行语义搜索
- 使用关键词查询进行精确匹配

**~~email：**
- 使用可用的邮件搜索工具
- 转换过滤器：`from:` 映射到发件人，日期映射到时间范围过滤器
- 根据情况将 `type:` 映射到附件过滤器或主题行搜索

**~~cloud storage：**
- 使用可用的文件搜索工具
- 转换为文件查询语法：名称包含、全文包含、修改日期、文件类型
- 同时考虑文件名和内容

**~~project tracker：**
- 使用可用的任务搜索或预输入工具
- 映射到任务文本搜索、负责人过滤器、日期过滤器、项目过滤器

**~~CRM：**
- 使用可用的 CRM 查询工具
- 跨账户、联系人、商机和其他相关对象搜索

**~~knowledge base：**
- 使用语义搜索处理概念性问题
- 使用关键词搜索进行精确匹配

### 4. 并行执行搜索

同时在所有可用来源运行所有子查询。不要等待一个来源完成再搜索另一个。

对于每个来源：
- 执行翻译后的查询
- 捕获带有元数据的结果（时间戳、作者、链接、来源类型）
- 记录任何失败或返回错误的来源——不要让一个失败阻止其他来源

### 5. 排序和去重结果

**去重：**
- 识别跨来源出现的相同信息（如在 ~~chat 和邮件中都提到的决定）
- 将相关结果归组，而非显示重复项
- 优先使用最权威或最完整的版本

**排序因素：**
- **相关性**：结果与查询意图的匹配程度
- **时效性**：状态/决定查询中更近期的结果排名更高
- **权威性**：事实性问题中官方文档 > Wiki > 聊天消息；"讨论了什么"查询中对话 > 文档
- **完整性**：背景更多的结果排名更高

### 6. 呈现统一结果

将响应格式化为合成答案，而非原始结果列表：

**对于事实/决定查询：**
```
[对问题的直接回答]

来源：
- [来源 1：简短描述]（~~chat，#频道，日期）
- [来源 2：简短描述]（~~email，来自某人，日期）
- [来源 3：简短描述]（~~cloud storage，文档名称，最后修改）
```

**对于探索性查询（"我们对 X 了解什么"）：**
```
[结合所有来源信息的合成摘要]

发现于：
- ~~chat：Y 个频道中 X 条相关消息
- ~~email：X 条相关讨论串
- ~~cloud storage：X 个相关文档
- [其他适用来源]

关键来源：
- [最重要来源及链接/引用]
- [第二重要来源]
```

**对于"查找"查询（寻找特定内容）：**
```
[他们要找的内容，带有直接引用]

还发现：
- [来自其他来源的相关条目]
```

### 7. 处理边缘情况

**模糊查询：**
如果查询可能有多种含义，搜索前先问一个澄清问题：
```
"API 重设计"可能指几种情况。您在找：
1. REST API v2 重设计（Aurora 项目）
2. 内部 SDK API 变更
3. 其他内容？
```

**无结果：**
```
我无法在 [已搜索来源列表] 中找到与"[查询]"匹配的内容。

尝试：
- 更宽泛的术语（如"database"而非"PostgreSQL migration"）
- 不同的时间范围（当前搜索[时间范围]）
- 检查相关来源是否已连接（当前搜索：[来源]）
```

**部分结果（某些来源失败）：**
```
[来自成功来源的结果]

注意：此次搜索无法访问 [失败来源]。
以上结果仅来自 [成功来源]。
```

## 说明

- 始终并行搜索多个来源——绝不顺序搜索
- 将结果合成为答案，不要仅列出原始搜索结果
- 包含来源归属，使用户可以深入查看
- 尊重用户的过滤器语法，并针对每个来源适当应用
- 当查询提到特定人员时，跨所有来源搜索其消息/文档/提及
- 对于时间敏感的查询，在排序中优先考虑时效性
- 如果只连接了一个来源，仍然提供该来源的有用结果
