# 知识合成

企业搜索的最后一公里。将来自多个来源的原始结果转化为连贯、可信赖的答案。

## 目标

将这个：
```
~~chat 结果："Sarah 在 #eng 中说：'我们用 REST，GraphQL 对我们的用例来说太复杂了'"
~~email 结果："主题：API 决定——Sarah 的邮件确认了 REST 方案及理由"
~~cloud storage 结果："API Design Doc v3——更新了第 2 节以反映 REST 决定"
~~project tracker 结果："任务：确定 API 方案——由 Sarah 标记完成"
```

转化为这个：
```
团队决定在 API 重设计中采用 REST 而非 GraphQL。Sarah 做出了
这个决定，指出 GraphQL 对当前用例来说过于复杂。这在周二的
#engineering 中讨论，周三通过邮件确认，设计文档已
更新以反映该决定。相关的 ~~project tracker 任务已标记完成。

来源：
- ~~chat: #engineering 讨论串（1 月 14 日）
- ~~email: 来自 Sarah 的"API 决定"（1 月 15 日）
- ~~cloud storage: "API Design Doc v3"（1 月 15 日更新）
- ~~project tracker: "确定 API 方案"（1 月 15 日完成）
```

## 去重

### 跨来源去重

相同信息经常出现在多个地方。识别并合并重复项：

**结果关于同一事物的信号：**
- 相同或非常相似的文本内容
- 相同的作者/发件人
- 时间戳在短时间窗口内（同一天或相邻天）
- 对相同实体的引用（项目名称、文档、决定）
- 一个来源引用另一个（"如 ~~chat 中讨论的"、"根据邮件"、"见文档"）

**如何合并：**
- 合并为单一叙述条目
- 引用所有出现的来源
- 使用最完整的版本作为主要文本
- 添加每个来源的独特细节

### 去重优先级

当相同信息存在于多个来源时，优先选择：
```
1. 最完整的版本（最充分的背景）
2. 最权威的来源（官方文档 > 聊天）
3. 最新版本（最新更新胜出）
```

### 不去重的情况

以下情况保留为独立条目：
- 讨论相同主题但结论不同时
- 不同的人表达不同观点时
- 信息在来源之间有实质性演变时（决定的 v1 与 v2）
- 代表不同时间段时

## 引用和来源归属

合成答案中的每个说法都必须可归属到来源。

### 归属格式

内联直接引用：
```
Sarah 在周三的邮件中确认了 REST 方案。
设计文档已更新以反映此决定（~~cloud storage: "API Design Doc v3"）。
```

末尾来源列表以保证完整性：
```
来源：
- ~~chat: #engineering 讨论（1 月 14 日）— 初始决定讨论串
- ~~email: 来自 Sarah Chen 的"API 决定"（1 月 15 日）— 正式确认
- ~~cloud storage: "API Design Doc v3"最后修改于 1 月 15 日 — 更新的规格
```

### 归属规则

- 始终注明来源类型（~~chat、~~email、~~cloud storage 等）
- 包含具体位置（频道、文件夹、讨论串）
- 包含日期或相对时间
- 相关时包含作者
- 可用时包含文档/讨论串标题
- 对于 ~~chat，注明频道名称
- 对于 ~~email，注明主题行和发件人
- 对于 ~~cloud storage，注明文档标题

## 置信度级别

并非所有结果都同样可信。根据以下因素评估置信度：

### 时效性

| 时效性 | 置信度影响 |
|---------|------------------|
| 今天 / 昨天 | 对当前状态高置信度 |
| 本周 | 良好置信度 |
| 本月 | 中等——情况可能已变化 |
| 超过一个月 | 较低置信度——标记为可能过时 |

对于状态查询，时效性权重很高。对于政策/事实查询，时效性重要性较低。

### 权威性

| 来源类型 | 权威级别 |
|-------------|----------------|
| 官方 Wiki / 知识库 | 最高——经过整理、维护 |
| 共享文档（最终版本） | 高——有意发布 |
| 邮件公告 | 高——正式沟通 |
| 会议纪要 | 中高——可能不完整 |
| 聊天消息（讨论串结论） | 中等——非正式但实时 |
| 聊天消息（讨论串中段） | 较低——可能不反映最终立场 |
| 草稿文档 | 低——未最终确定 |
| 任务评论 | 取决于评论者 |

### 表达置信度

置信度高时（多个新鲜、权威来源一致）：
```
团队决定在 API 重设计中使用 REST。[直接陈述]
```

置信度中等时（单一来源或稍旧）：
```
根据上个月 #engineering 中的讨论，团队倾向于
在 API 重设计中使用 REST。此后情况可能有所变化。
```

置信度低时（旧数据、非正式来源或信号冲突）：
```
我在 ~~chat 中找到了三个月前关于 API 迁移讨论的引用，
但找不到正式的决定文档。这些信息可能已过时。
您可能需要向团队确认当前状态。
```

### 信息冲突

当来源不一致时：
```
我发现关于 API 方案的信息相互冲突：
- 1 月 10 日的 ~~chat 讨论建议使用 GraphQL
- 但 Sarah 1 月 15 日的邮件确认了使用 REST
- 设计文档（1 月 15 日更新）反映了 REST

最新来源表明 REST 是最终决定，
但早期的 ~~chat 讨论先探讨了 GraphQL。
```

始终呈现冲突，而非默默地选择一个版本。

## 摘要策略

### 对于小结果集（1-5 条结果）

呈现每条结果及背景。不需要摘要——给用户所有内容：
```
[从结果合成的直接答案]

[来源 1 的细节]
[来源 2 的细节]

来源：[完整归属]
```

### 对于中等结果集（5-15 条结果）

按主题分组并摘要每组：
```
[总体答案]

主题 1：[相关结果摘要]
主题 2：[相关结果摘要]

关键来源：[前 3-5 个最相关来源]
完整结果：跨 [来源] 找到 [数量] 条
```

### 对于大结果集（15 条以上结果）

提供高层次合成，并提供深入探索的选项：
```
[基于最相关结果的总体答案]

摘要：
- [关键发现 1]（由 N 个来源支持）
- [关键发现 2]（由 N 个来源支持）
- [关键发现 3]（由 N 个来源支持）

主要来源：
- [最权威/相关的来源]
- [第二相关来源]
- [第三相关来源]

跨 [来源列表] 共找到 [总数] 条结果。
需要我深入探讨某个具体方面吗？
```

### 摘要规则

- 以答案开头，而非搜索过程
- 不要列出原始结果——将其合成为叙述
- 将来自不同来源的相关条目归组
- 保留重要的细微差别和注意事项
- 包含足够的细节，使用户可以决定是否深入
- 当结果集较大时，始终提供提供更多细节的选项

## 合成工作流

```
[来自所有来源的原始结果]
          ↓
[1. 去重——合并来自不同来源的相同信息]
          ↓
[2. 聚类——按主题/话题将相关结果分组]
          ↓
[3. 排序——按与查询的相关性对聚类和条目排序]
          ↓
[4. 评估置信度——时效性 × 权威性 × 一致性]
          ↓
[5. 合成——生成带归属的叙述性答案]
          ↓
[6. 格式化——根据结果数量选择适当的详细程度]
          ↓
[带来源的连贯答案]
```

## 反模式

**不要：**
- 按来源逐一列出结果（"来自 ~~chat：... 来自 ~~email：... 来自 ~~cloud storage：..."）
- 仅因为匹配关键词就包含不相关结果
- 将答案埋在方法论说明之下
- 呈现相互冲突的信息而不标记冲突
- 省略来源归属
- 以与有据可查的事实相同的置信度呈现不确定信息
- 过度摘要导致有用细节丢失

**要：**
- 以答案开头
- 按主题分组，而非按来源
- 适当时标记置信度级别
- 明确呈现冲突
- 为所有说法归属来源
- 当结果集较大时，提供深入探讨的选项
