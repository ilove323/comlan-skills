# /customer-research

> 如果遇到不熟悉的占位符或需要查看已连接的工具，请参阅 [CONNECTORS.md](../../CONNECTORS.md)。

对客户问题、产品话题或账户相关查询进行多来源调研。整合所有可用来源的发现，提供清晰的来源标注和置信度评分。

## 用法

```
/customer-research <question or topic>
```

## 工作流程

### 1. 解析调研请求

识别所需调研的类型：
- **客户问题**：客户提出的需要解答的问题（例如："我们的产品支持 Okta SSO 吗？"）
- **问题调查**：已报告问题的背景（例如："这个 Bug 以前被报告过吗？已知的变通方案是什么？"）
- **账户背景**：与特定客户的历史（例如："上次 Acme Corp 问这个问题时，我们告诉他们什么？"）
- **话题调研**：与支持工作相关的一般话题（例如："Webhook 重试逻辑的最佳实践"）

搜索前，明确您实际要找的内容：
- 这是有明确答案的事实性问题吗？
- 这是需要多方视角的情境性问题吗？
- 这是范围尚未确定的探索性问题吗？
- 答案的受众是谁（内部团队、客户、管理层）？

### 2. 搜索可用来源

根据以下来源层级系统地搜索，适配已连接的工具。不要在第一个结果就停下——要跨来源交叉验证。

**第一层——官方内部来源（最高置信度）：**
- ~~knowledge base（如已连接）：产品文档、操作手册、FAQ、政策文件
- ~~cloud storage：内部文档、规格说明、指南、过往调研
- 产品路线图（内部面向）：功能时间线、优先级

**第二层——组织背景：**
- ~~CRM 备注：账户备注、活动历史、过往答复、商机详情
- ~~support platform（如已连接）：过往解决方案、已知问题、变通方案
- 会议记录：过往讨论、决策、承诺

**第三层——团队沟通：**
- ~~chat：在相关频道搜索话题；查看同事是否讨论过或回答过这个问题
- ~~email：搜索关于此话题的过往往来邮件
- 日历备注：会议议程和会后笔记

**第四层——外部来源：**
- 网络搜索：官方文档、博客文章、社区论坛
- 公开知识库、帮助中心、发布说明
- 第三方文档：集成合作伙伴、互补工具

**第五层——推断或类比（当直接来源没有答案时使用）：**
- 类似情况：类似问题过去是如何处理的
- 类似客户：对可比账户有效的方法
- 通用最佳实践：行业标准和规范

### 3. 综合发现

将结果汇编为结构化调研简报：

```
## Research: [Question/Topic]

### Answer
[Clear, direct answer to the question — lead with the bottom line]

**Confidence:** [High / Medium / Low]
[Explain what drives the confidence level]

### Key Findings

**From [Source 1]:**
- [Finding with specific detail]
- [Finding with specific detail]

**From [Source 2]:**
- [Finding with specific detail]

### Context & Nuance
[Any caveats, edge cases, or additional context that matters]

### Sources
1. [Source name/link] — [what it contributed]
2. [Source name/link] — [what it contributed]
3. [Source name/link] — [what it contributed]

### Gaps & Unknowns
- [What couldn't be confirmed]
- [What might need verification from a subject matter expert]

### Recommended Next Steps
- [Action if the answer needs to go to a customer]
- [Action if further research is needed]
- [Who to consult for verification if needed]
```

### 4. 处理来源不足的情况

如果已连接来源没有结果：

- 对该话题进行网络调研
- 向用户询问内部背景：
  - "我在已连接来源中找不到这个信息。您有关于此话题的内部文档或知识库文章吗？"
  - "您的团队以前讨论过这个话题吗？有什么 ~~chat 频道我应该检查吗？"
  - "有主题专家知道答案吗？"
- 对局限性保持透明：
  - "此答案仅基于网络调研——请在与客户分享前对照内部文档验证。"
  - "我找到了一个可能的答案，但无法从权威的内部来源确认。"

### 5. 面向客户的考量

如果调研是为了回答客户问题：

- 如果答案涉及产品路线图、定价、法律或安全话题，请标记并提示可能需要审查
- 如果答案与之前传达的内容不同，请注明
- 为面向客户的回复建议适当的注意事项
- 提议起草客户回复："要我根据这些发现起草一封客户回复吗？"

### 6. 知识沉淀

调研完成后，建议沉淀知识：

- "需要我将这些发现保存到知识库供将来参考吗？"
- "需要我根据此次调研创建一个 FAQ 条目吗？"
- "这可能值得记录下来——需要我起草一个操作手册条目吗？"

这有助于积累机构知识，减少团队中的重复调研工作。

## 来源优先级和置信度

### 按来源层级的置信度

| 层级 | 来源类型 | 置信度 | 备注 |
|------|---------|--------|------|
| 1 | 官方内部文档、知识库、政策 | **高** | 除非明显过时，否则信任——检查日期 |
| 2 | CRM、支持工单、会议记录 | **中高** | 可能主观或不完整 |
| 3 | 即时通讯、邮件、日历备注 | **中** | 非正式，可能断章取义或带推测 |
| 4 | 网络、论坛、第三方文档 | **中低** | 可能不反映您的具体情况 |
| 5 | 推断、类比、最佳实践 | **低** | 明确标注为推断而非事实 |

### 置信度级别

始终分配并传达置信度级别：

**高置信度：**
- 由官方文档或权威来源确认
- 多个来源印证相同答案
- 信息是最新的（在合理时间范围内验证过）
- "我对此非常有把握，依据是 [来源]。"

**中置信度：**
- 答案见于非正式来源（即时通讯、邮件）但非官方文档
- 单一来源，缺乏其他印证
- 信息可能略有过时但大概仍然有效
- "根据 [来源]，情况似乎如此，但建议向 [团队/人员] 确认。"

**低置信度：**
- 答案是从相关信息推断的
- 来源已过时或可能不可靠
- 各来源中存在矛盾信息
- "我没能找到明确答案。根据 [背景]，我的最佳判断是 [答案]，但在向客户分享前应予以验证。"

**无法确定：**
- 在任何来源中都未找到相关信息
- 问题需要来源中没有的专业知识
- "我找不到关于此问题的信息。建议联系 [建议的专家/团队] 获取明确答案。"

### 处理矛盾

当来源信息不一致时：
1. 明确指出矛盾
2. 确定哪个来源更权威或更新
3. 提供两种观点及背景
4. 建议如何解决分歧
5. 如果要告知客户：在解决前使用最保守/谨慎的答案

## 何时直接回答 vs. 升级验证

### 直接回答的情况：
- 官方文档清楚地解答了问题
- 多个可靠来源印证了答案
- 问题是事实性的且不敏感
- 答案不涉及承诺、时间线或定价
- 您之前回答过类似问题且准确性已得到确认

### 升级或验证的情况：
- 答案涉及产品路线图承诺或时间线
- 定价、法律条款或合同特定问题
- 安全、合规或数据处理问题
- 答案可能设立先例或产生预期
- 来源中存在矛盾信息
- 问题涉及特定客户的自定义配置
- 答案需要您没有的专业知识
- 客户面临风险，且错误答案可能加剧情况

### 升级路径：
1. **主题专家**：技术或专业领域问题
2. **产品团队**：路线图、功能或能力问题
3. **法律/合规**：条款、隐私、安全或监管问题
4. **财务/计费**：定价、发票或付款相关问题
5. **工程**：自定义配置、Bug 或技术根因
6. **管理层**：战略决策、例外情况或高风险情况

## 团队知识库的调研记录

调研完成后，为将来使用沉淀知识。

### 何时记录：
- 问题以前出现过或可能再次出现
- 调研需要大量时间汇编
- 答案需要整合多个来源
- 答案纠正了常见误解
- 答案涉及容易出错的细微差别

### 记录格式：
```
## [Question/Topic]

**Last Verified:** [date]
**Confidence:** [level]

### Answer
[Clear, direct answer]

### Details
[Supporting detail, context, and nuance]

### Sources
[Where this information came from]

### Related Questions
[Other questions this might help answer]

### Review Notes
[When to re-verify, what might change this answer]
```

### 知识库卫生：
- 为所有条目加时间戳
- 标注引用特定产品版本或功能的条目
- 每季度审阅和更新条目
- 归档不再相关的条目
- 为条目添加标签以便搜索（按话题、产品区域、客户群体）
