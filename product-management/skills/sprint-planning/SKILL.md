# /sprint-planning

> 如遇到不熟悉的占位符或需要查看哪些工具已接入，请参阅 [CONNECTORS.md](../../CONNECTORS.md)。

通过界定工作范围、估算产能和设定清晰目标来规划迭代。

## 使用方式

```
/sprint-planning $ARGUMENTS
```

## 工作原理

```
┌─────────────────────────────────────────────────────────────────┐
│                    迭代规划                                        │
├─────────────────────────────────────────────────────────────────┤
│  单独使用（始终有效）                                              │
│  ✓ 定义迭代目标和成功标准                                          │
│  ✓ 估算团队产能（考虑带薪休假、会议）                              │
│  ✓ 界定和排序待办条目                                              │
│  ✓ 识别依赖项和风险                                               │
│  ✓ 生成迭代计划文档                                               │
├─────────────────────────────────────────────────────────────────┤
│  工具增强（接入工具后）                                            │
│  + 项目追踪：拉取待办列表，创建迭代，分配条目                      │
│  + 日历：在产能计算中考虑带薪休假和会议                            │
│  + 沟通工具：与团队分享迭代计划                                    │
└─────────────────────────────────────────────────────────────────┘
```

## 我需要的信息

- **团队**：谁在团队中，本迭代的可用性如何？
- **迭代时长**：多少天/周？
- **待办列表**：哪些已排优先级？（从追踪工具获取、粘贴或描述）
- **遗留工作**：上个迭代有未完成的吗？
- **依赖项**：是否有被其他团队阻塞的事项？

## 提示

1. **留出缓冲** — 按 70-80% 产能规划。总会有计划外的事情。
2. **一个清晰的迭代目标** — 如果无法用一句话表述，迭代就缺乏聚焦。
3. **确定冲刺条目** — 提前知道如果事情超时应该削减什么。
4. **诚实对待遗留工作** — 如果某件事没有交付，先理解原因，再重新承诺。
