# 撰写规格文档

> 如遇到不熟悉的占位符或需要查看哪些工具已接入，请参阅 [CONNECTORS.md](../../CONNECTORS.md)。

撰写功能规格文档或产品需求文档（PRD）。

## 使用方式

```
/write-spec $ARGUMENTS
```

## 工作流

### 1. 理解功能需求

询问用户想要为什么写规格文档。接受以下任意形式：
- 功能名称（"SSO 支持"）
- 问题陈述（"企业客户一直在要求集中式认证"）
- 用户需求（"用户希望以 CSV 格式导出数据"）
- 模糊想法（"我们应该对入门流失率做些什么"）

### 2. 收集背景信息

向用户询问以下内容。保持对话感——不要一次抛出所有问题。先问最重要的，边推进边补充细节：

- **用户问题**：这解决了什么问题？谁在经历它？
- **目标用户**：这服务于哪个用户群体？
- **成功指标**：我们如何知道这有效了？
- **约束条件**：技术约束、时间线、法规要求、依赖关系
- **历史沿革**：之前是否尝试过？是否有现有解决方案？

### 3. 从已接入工具提取背景

如果 **~~project tracker** 已接入：
- 搜索相关工单、史诗或功能
- 提取已有的需求或验收标准
- 识别对其他工作项的依赖

如果 **~~knowledge base** 已接入：
- 搜索相关研究文档、历史规格或设计文档
- 提取相关用户研究发现
- 查找相关会议记录或决策记录

如果 **~~design** 已接入：
- 拉取相关原型图、线框图或设计探索
- 搜索与功能相关的设计系统组件

如果这些工具没有接入，完全基于用户提供的信息工作。不要要求用户去接入工具——直接用现有信息推进。

### 4. 生成 PRD

按以下章节产出结构化 PRD。每个章节的详细指导见下方**PRD 结构**。

- **问题陈述**：用户问题、影响对象，以及不解决的后果（2-3 句话）
- **目标**：3-5 个与用户或业务指标挂钩的具体、可衡量成果
- **非目标**：3-5 件明确不在范围内的事，每条附带简短原因
- **用户故事**：标准格式（"作为[用户类型]，我希望[能力]，以便[收益]"），按角色分组
- **需求**：分为必须有（P0）、建议有（P1）和未来考量（P2），每条附带验收标准
- **成功指标**：领先指标（快速变化）和滞后指标（随时间变化），包含具体目标值
- **待解决问题**：未解答的问题，标注谁需要回答（工程、设计、法务、数据）
- **时间线考量**：硬性截止日期、依赖项和分阶段建议

### 5. 审阅与迭代

生成 PRD 后：
- 询问用户是否需要调整某些章节
- 提供对特定章节的扩展
- 提供创建后续产出物的服务（设计需求说明、工程工单拆解、干系人提案）

## PRD 结构

### 问题陈述
- 用 2-3 句话描述用户问题
- 谁在经历这个问题，多频繁
- 不解决的代价是什么（用户痛点、业务影响、竞争风险）
- 以证据为基础：用户研究、支持数据、指标或客户反馈

### 目标
- 3-5 个这个功能应该实现的具体、可衡量成果
- 每个目标应该回答："我们如何知道这成功了？"
- 区分用户目标（用户获得什么）和业务目标（公司获得什么）
- 目标应该是结果，而非输出（"将首次获得价值的时间缩短 50%"，而非"构建入门向导"）

### 非目标
- 3-5 件这个功能明确不会做的事
- 本次版本范围之外的相邻能力
- 对每个非目标，简要解释为什么不在范围内（影响不够大、过于复杂、独立计划、为时过早）
- 非目标在实施过程中防止范围蔓延，并向干系人设定预期

### 用户故事
用标准格式撰写用户故事："作为[用户类型]，我希望[能力]，以便[收益]"

指导原则：
- 用户类型应该具体到有意义（"企业管理员"，而不只是"用户"）
- 能力应该描述他们想要完成什么，而不是怎么做
- 收益应该解释"为什么"——这带来了什么价值
- 包括边界情况：错误状态、空状态、边界条件
- 如果功能服务于多个角色，包含不同的用户类型
- 按优先级排序——最重要的故事在前

示例：
- "作为团队管理员，我希望为我的组织配置 SSO，以便我的团队成员可以使用企业凭证登录"
- "作为团队成员，我希望被自动重定向到公司的 SSO 登录，以便我不需要记住单独的密码"
- "作为团队管理员，我希望查看哪些成员通过 SSO 登录，以便我可以验证推广是否正常工作"

### 需求

**必须有（P0）**：没有这些，功能就无法发布。这些代表功能的最低可行版本。问："如果我们去掉这个，功能还能解决核心问题吗？"如果不能，它就是 P0。

**建议有（P1）**：显著改善体验，但没有这些核心使用场景也能工作。这些通常在发布后快速跟进。

**未来考量（P2）**：明确不在 v1 范围内，但我们希望以支持它们的方式设计。记录这些可以防止意外的架构决策使它们难以在将来实现。

对每个需求：
- 清晰、无歧义地描述预期行为
- 包含验收标准（见下文）
- 注明任何技术考量或约束
- 标记对其他团队或系统的依赖

### 待解决问题
- 在实施之前或期间需要回答的问题
- 每条标注谁应该回答（工程、设计、法务、数据、干系人）
- 区分阻断性问题（开始前必须回答）和非阻断性问题（实施过程中可以解决）

### 时间线考量
- 硬性截止日期（合同承诺、活动、合规日期）
- 对其他团队工作或发布的依赖
- 如果功能太大无法一次发布，建议的分阶段方案

## 用户故事撰写

好的用户故事是：
- **独立的**：可以独立开发和交付
- **可协商的**：细节可以讨论，故事不是合同
- **有价值的**：为用户（而不只是团队）带来价值
- **可估算的**：团队可以粗略估算工作量
- **小的**：可以在一个迭代/周期内完成
- **可测试的**：有明确的验证方式

### 用户故事的常见错误
- 太模糊："作为用户，我希望产品更快"——具体什么应该更快？
- 规定解决方案："作为用户，我希望有一个下拉菜单"——描述需求，而不是 UI 组件
- 没有收益："作为用户，我希望点击一个按钮"——为什么？它完成了什么？
- 太大："作为用户，我希望管理我的团队"——拆解为具体能力
- 内部视角："作为工程团队，我们希望重构数据库"——这是任务，不是用户故事

## 需求分类

### MoSCoW 框架
- **必须有（Must have）**：没有这些，功能就不可行。不可协商。
- **应该有（Should have）**：重要但对发布不是关键。高优先级的快速跟进。
- **可以有（Could have）**：如果时间允许值得有。不会因此延迟交付。
- **这次不做（Won't have this time）**：明确不在范围内。可能在未来版本中重新考虑。

### 分类提示
- 对 P0 要严格。必须有的列表越精简，发布越快，学习越早。
- 如果所有东西都是 P0，那就没有什么是 P0。挑战每一个"必须有"："我们真的不发布这个吗？"
- P1 应该是你有把握很快会构建的东西，而不是心愿单。
- P2 是架构保险——它们引导设计决策，即使你现在不构建它们。

## 成功指标定义

### 领先指标
发布后快速变化的指标（几天到几周）：
- **采用率**：有资格使用的用户中实际尝试功能的比例
- **激活率**：完成核心动作的用户比例
- **任务完成率**：成功完成目标的用户比例
- **完成时间**：核心工作流需要多长时间
- **错误率**：用户遇到错误或死胡同的频率
- **功能使用频率**：用户回来使用功能的频率

### 滞后指标
需要时间才能显现的指标（几周到几个月）：
- **留存影响**：这个功能是否改善了用户留存？
- **营收影响**：这是否带动了升级、扩张或新营收？
- **NPS/满意度变化**：这是否改善了用户对产品的感受？
- **支持工单减少**：这是否降低了支持负担？
- **竞争胜率**：这是否帮助赢得更多交易？

### 设定目标值
- 目标应该具体："30 天内 50% 的采用率"，而非"高采用率"
- 基于类似功能、行业基准或明确的假设来设定目标
- 设定"成功"阈值和"冲刺"目标
- 定义衡量方法：什么工具、什么查询、什么时间窗口
- 指定何时评估：发布后 1 周、1 月、1 季度

## 验收标准

用 Given/When/Then 格式或清单形式撰写验收标准：

**Given/When/Then 格式**：
- 假设 [前置条件或情境]
- 当 [用户采取的动作]
- 则 [预期结果]

示例：
- 假设管理员已为其组织配置了 SSO
- 当团队成员访问登录页面时
- 则他们将自动重定向到该组织的 SSO 提供商

**清单格式**：
- [ ] 管理员可以在组织设置中输入 SSO 提供商 URL
- [ ] 团队成员在登录页看到"通过 SSO 登录"按钮
- [ ] SSO 登录在账号不存在时创建新账号
- [ ] SSO 登录在邮箱匹配时关联现有账号
- [ ] SSO 登录失败时显示清晰的错误信息

### 验收标准提示
- 覆盖正常路径、错误情况和边界条件
- 具体说明预期行为，而非实现方式
- 包含不应该发生的内容（负向测试用例）
- 每条标准应该能独立测试
- 避免模糊词汇："快速"、"友好"、"直觉化"——具体说明这些意味着什么

## 范围管理

### 识别范围蔓延
范围蔓延发生在以下情况：
- 规格文档批准后，需求不断被追加
- "小"改动积累成显著更大的项目
- 团队在构建没有用户要求的功能（"既然做了……"）
- 发布日期不断推移却没有明确的重新界定范围
- 干系人追加需求却不去掉任何东西

### 防止范围蔓延
- 在每份规格文档中写明确的非目标
- 要求任何范围增加都伴随着范围移除或时间线延期
- 在规格文档中清晰区分"v1"和"v2"
- 对照原始问题陈述审查规格文档——所有内容都在服务于它吗？
- 对调研设置时间盒："如果 2 天内搞不清楚 X，就砍掉它"
- 为好想法但不在范围内的内容创建"暂停区"

## 输出格式

使用带清晰标题的 Markdown 格式。让文档易于扫描——繁忙的干系人应该能够只读标题和加粗文字就能抓住要点。

## 提示

- 对范围要有主见。一个精简、定义清晰的规格文档，好过一个宏大模糊的规格文档。
- 如果用户的想法太大无法用一份规格文档覆盖，建议分阶段，并为第一阶段写规格。
- 成功指标应该具体且可衡量，而不是模糊的（"改善用户体验"）。
- 非目标和目标同样重要。它们在实施过程中防止范围蔓延。
- 待解决问题应该是真正待解决的——不要包含你已经能从背景中回答的问题。
