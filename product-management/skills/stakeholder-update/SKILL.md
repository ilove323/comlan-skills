# 干系人更新

> 如遇到不熟悉的占位符或需要查看哪些工具已接入，请参阅 [CONNECTORS.md](../../CONNECTORS.md)。

生成针对受众和节奏定制的干系人更新。

## 使用方式

```
/stakeholder-update $ARGUMENTS
```

## 工作流

### 1. 确定更新类型

询问用户是哪种更新：
- **周报**：常规节奏的进展、阻碍和后续步骤更新
- **月报**：更宏观的趋势、里程碑和战略对齐摘要
- **发布公告**：功能或产品发布的公告，包含细节和影响
- **临时更新**：特定情境的一次性更新（上报、转向、重大决策）

### 2. 确定受众

询问更新发给谁：
- **高管/管理层**：高层次、以结果为导向、战略框架、简洁
- **工程团队**：技术细节、实施背景、阻碍、待决策事项
- **跨职能合作伙伴**：适当的细节、聚焦共同目标和依赖关系
- **客户/外部**：以收益为框架、时间线清晰、无内部术语
- **董事会**：以指标为驱动、战略层面、风险聚焦、非常简洁

### 3. 从已接入工具获取上下文

如果 **~~project tracker** 已接入：
- 拉取路线图条目和里程碑的状态
- 识别自上次更新以来的已完成条目
- 标出高风险或受阻条目
- 拉取迭代或周期进展

如果 **~~chat** 已接入：
- 搜索相关团队讨论和决策
- 查找频道中提到的阻碍或问题
- 识别异步做出的关键决策

如果 **~~meeting transcription** 已接入：
- 拉取近期会议记录和讨论摘要
- 从相关会议中找出决策和待办事项

如果 **~~knowledge base** 已接入：
- 搜索近期会议记录
- 查找决策文档或设计评审

如果没有接入任何工具，请用户提供：
- 自上次更新以来完成了什么
- 当前的阻碍或风险
- 已做出或待做出的关键决策
- 接下来要做什么

### 4. 生成更新

使用下方的模板和框架，为目标受众撰写结构化更新。

**对高管**：一句话摘要、状态颜色（绿/黄/红）、与目标挂钩的核心进展、已做决策、风险及缓解方案、具体诉求和下一里程碑。控制在 300 字以内。

**对工程团队**：已交付内容（含链接）、进行中内容（含负责人）、阻碍、待决策事项（含选项和建议）以及后续计划。

**对跨职能合作伙伴**：即将影响他们的内容、需要他们做的事（含截止日期）、影响其团队的决策，以及可以提供意见的领域。

**对客户**：新功能（以收益为框架）、即将上线的功能、已知问题及解决方案和反馈渠道。无内部术语。

**对发布公告**：已上线的内容、为什么重要、关键细节（范围、可用性、限制）、成功指标、发布计划和反馈渠道。

### 5. 审阅与交付

生成更新后：
- 询问用户是否需要调整语气、详细程度或重点
- 提供针对交付渠道（邮件、聊天帖、文档、幻灯片）的格式
- 如果 **~~chat** 已接入，提供起草发送消息的服务

## 各受众更新模板

### 高管/管理层更新
高管想要：战略背景、对目标的进展、需要他们帮助解决的风险、需要他们意见的决策。

**格式**：
```
状态：[绿 / 黄 / 红]

一句话摘要：[一句话——最重要的事情]

进展：
- [以目标/OKR 表述的已实现成果]
- [已达里程碑，及其影响]
- [核心指标变动]

风险：
- [风险]：[缓解方案]。[如需帮助，请写明诉求]。

待决策：
- [决策]：[选项及建议]。需要在 [日期] 前确定。

下一里程碑：
- [里程碑] — [日期]
```

**高管更新提示**：
- 先说结论，再讲过程。高管想听"我们交付了 X，推动了 Y 指标"，而不是"我们开了 14 次站会，关闭了 23 个工单"。
- 控制在 200 字以内。如果他们想了解更多，他们会问的。
- 状态颜色应该反映你的真实评估，而不是你认为他们想听的。黄色不是失败——那是良好的风险管理。
- 只包含你需要帮助解决的风险。除非他们需要知道，否则不要列出你已经在处理的风险。
- 诉求必须具体："周五前在 X 上做出决定"，而非"需要支持"。

### 工程团队更新
工程师想要：明确的优先级、技术背景、已解决的阻碍、影响他们工作的决策。

**格式**：
```
已交付：
- [功能/修复] — [PR/工单链接]。[如有值得一提的影响]。

进行中：
- [条目] — [负责人]。[预计完成时间]。[如有阻碍]。

决策：
- [已做决策]：[理由]。[ADR 链接如有]。
- [待做决策]：[背景]。[选项]。[建议]。

优先级变更：
- [什么变了，为什么]

后续计划：
- [下一批条目] — [为什么是这些]
```

**工程团队更新提示**：
- 链接到具体工单、PR 和文档。工程师想要点击链接了解详情。
- 优先级变化时，解释原因。当工程师理解原因时，认同感更强。
- 明确说明什么在阻碍他们，以及你在做什么来解除阻碍。
- 不要用与他们工作无关的信息浪费他们的时间。

### 跨职能合作伙伴更新
合作伙伴（设计、市场、销售、支持）想要：即将影响他们的内容、他们需要做哪些准备、如何提供意见。

**格式**：
```
即将上线：
- [功能/发布] — [日期]。[这对你们团队意味着什么]。

需要你们配合的：
- [具体诉求] — [背景]。截止 [日期]。

已做决策：
- [决策] — [如何影响你们团队]。

可以提供意见的：
- [我们希望收到反馈的话题] — [如何提供反馈]。
```

### 客户/外部更新
客户想要：新增功能、即将上线的内容、对他们的收益、如何上手使用。

**格式**：
```
新增内容：
- [功能] — [以客户语言描述的收益]。[如何使用/链接]。

即将上线：
- [功能] — [预计时间]。[为何对你重要]。

已知问题：
- [问题] — [状态]。[如有解决方案]。

反馈：
- [如何分享反馈或提交需求]
```

**客户更新提示**：
- 无内部术语。无工单号。无技术实现细节。
- 所有内容以客户现在能做什么来表述，而非你做了什么。
- 对时间线诚实，但不要过度承诺。"本季度晚些时候"好过一个可能错过的具体日期。
- 只提及客户可感知的已知问题，以及有解决方案的情况。

## 状态报告框架

### 绿 / 黄 / 红 状态

**绿**（达标）：
- 按计划进展
- 没有重大风险或阻碍
- 有望按时完成承诺
- 当事情真的进展顺利时才用绿色——不要作为默认状态

**黄**（高风险）：
- 进展慢于计划，或风险已经显现
- 缓解措施正在推进，但结果不确定
- 如果不干预或不调整范围，可能会错过承诺
- 主动使用黄色——越早标记风险，你的选项越多

**红**（偏轨）：
- 显著落后于计划
- 重大阻碍或风险，没有明确的缓解方案
- 如果没有重大干预（削减范围、增加资源、延期），将错过承诺
- 真正需要帮助时才用红色。不要等到太晚。

### 何时切换状态
- 出现风险迹象时立即切换到黄色，而不是确认情况已经很糟糕时
- 当你穷尽了自己的选项、需要上报时切换到红色
- 只有在风险真正解决后才切换回绿色，而不是只是暂停了
- 切换状态时记录变化原因——"切换到黄色，原因是 [理由]"

## 风险沟通

### ROAM 风险管理框架
- **已解决（Resolved）**：风险不再是问题。记录如何解决的。
- **已认领（Owned）**：风险已知悉，有人在积极管理。说明负责人和缓解方案。
- **已接受（Accepted）**：风险已知，但我们选择在没有缓解措施的情况下继续。记录原因。
- **已缓解（Mitigated）**：已采取行动将风险降至可接受水平。记录做了什么。

### 有效的风险沟通
1. **清晰陈述风险**："有风险 [某事] 发生，因为 [原因]"
2. **量化影响**："如果发生，后果是 [影响]"
3. **说明可能性**："这是 [可能/可能/不太可能] 的，因为 [证据]"
4. **给出缓解方案**："我们正在通过 [行动] 来管理这个风险"
5. **提出诉求**："我们需要 [具体帮助] 来进一步降低这个风险"

### 风险沟通的常见错误
- 把风险埋在好消息里。当风险重要时，先说风险。
- 含糊其辞："可能会有一些延期"——具体说明什么、多长时间、为什么。
- 提出风险但没有缓解方案。每个风险都应该附带一个方案。
- 等太久。提前沟通的风险是规划输入；延迟沟通的风险是救火演习。

## 决策记录（ADR）

### 架构决策记录格式
记录重要决策以供将来参考：

```
# [决策标题]

## 状态
[已提议 / 已接受 / 已废弃 / 被 ADR-XXX 取代]

## 背景
是什么情境需要做出这个决策？存在哪些制约因素？

## 决策
我们决定了什么？清晰直接地陈述决策。

## 后果
这个决策的含义是什么？
- 正面后果
- 负面后果或已接受的取舍
- 这会带来什么，又会关闭什么

## 已考虑的替代方案
评估了哪些其他选项？
对每个选项：它是什么，为什么被否决？
```

### 何时撰写 ADR
- 战略性产品决策（目标哪个市场细分、支持哪个平台）
- 重大技术决策（架构选择、供应商选择、自建 vs 采购）
- 有争议的决策，人们持有分歧（为将来记录决策依据）
- 限制未来选项的决策（选择技术、签署合作协议）
- 预期将来会被质疑的决策（趁背景还新鲜时记录下来）

### 决策记录提示
- 在决策做出后尽快撰写 ADR，而不是几周后
- 记录谁参与了决策以及最终谁拍板
- 慷慨地记录背景——未来的读者不会拥有今天的背景
- 记录事后看起来是错误的决策也是可以的——添加"被 XXX 取代"链接
- 保持简短。一页比五页更好。

## 会议引导

### 站会/日会
**目的**：暴露阻碍，协调工作，保持动力。
**格式**：每人分享：
- 自上次同步以来完成了什么
- 接下来要做什么
- 是什么在阻碍他们

**引导提示**：
- 控制在 15 分钟以内。如果出现讨论，单独处理。
- 聚焦阻碍——这是站会最有价值的部分
- 追踪阻碍并跟进解决情况
- 如果没有什么需要同步的，取消站会。尊重大家的时间。

### 迭代/周期规划
**目的**：承诺下一迭代的工作；对齐优先级和范围。
**格式**：
1. 复盘：上个迭代交付了什么，遗留了什么，削减了什么
2. 优先级：本迭代最重要的事情是什么
3. 产能：团队能承担多少（考虑带薪休假、值班、会议）
4. 承诺：从待办列表中选择符合产能和优先级的条目
5. 依赖项：标记跨团队或外部依赖

**引导提示**：
- 带着建议的优先顺序来。不要从零开始让团队排序。
- 对过度承诺说"不"。承诺少、稳定交付，好过承诺多、令人失望。
- 确保每个条目都有明确的负责人和清晰的验收标准。
- 标记出定义不清或存在隐藏复杂性的条目。

### 回顾
**目的**：反思什么做得好，什么做得不好，以及需要改变什么。
**格式**：
1. 营造氛围：提醒团队目的，建立心理安全
2. 收集数据：做得好的、做得不好的、令人困惑的
3. 生成洞察：识别模式和根本原因
4. 决定行动：挑选 1-3 个具体的改善措施，在下个迭代尝试
5. 收尾：感谢大家的坦诚反馈

**引导提示**：
- 建立心理安全感。人们必须感到可以安全地诚实表达。
- 聚焦系统和流程，而非个人。
- 限制在 1-3 个行动项。超过这个数量什么都不会改变。
- 跟进上一次回顾的行动项。如果从不跟进，人们会停止参与。
- 偶尔变换回顾格式以防止僵化。

### 干系人评审/演示
**目的**：展示进展，收集反馈，建立对齐。
**格式**：
1. 背景：提醒干系人目标，以及他们上次看到的是什么
2. 演示：展示已构建的内容。使用真实产品，不要用幻灯片。
3. 指标：分享早期数据或反馈
4. 反馈：结构化的问答和意见收集时间
5. 后续步骤：接下来什么，以及下次评审的时间

**引导提示**：
- 尽可能演示真实产品。幻灯片不是演示。
- 明确反馈收集的框架："你对 X 有什么反馈？"好过"有什么想法？"
- 可见地记录反馈，并承诺响应它（或解释为什么不做）
- 设定关于在这个阶段哪类反馈是可行动的预期

## 输出格式

保持更新内容易于扫描。关键点加粗，列表用项目符号。高管更新控制在 300 字以内。工程更新可以更长，但仍应结构清晰以支持快速浏览。

## 提示

- 干系人更新中最常见的错误是把重点埋起来。从最重要的事情开始。
- 状态颜色（绿/黄/红）应该反映现实，而非乐观预期。黄色不是失败——那是良好的风险沟通。
- 诉求必须具体且可行动。"我们需要帮助"不是诉求。"我们需要在周五前对 X 做出决定"才是。
- 对高管，所有内容都以成果和目标框架呈现，而非活动和任务。
- 如果有坏消息，先说坏消息。不要把它藏在好消息之后。
- 长度要匹配受众的注意力。高管几条要点就够了；工程团队需要他们所需的细节。
